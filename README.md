## 1. 專案概述

### 1.1 系統定位

Horizon Nova 是一個功能完整的企業級後台管理系統，提供全方位的管理工具與權限控制功能，包含使用者管理、角色管理、組織管理、檔案管理、系統監控、AI 功能等。

**系統需求背景：**

企業需要一個統一的後台管理系統來管理組織架構、使用者權限、系統資源等。傳統的多系統分散管理方式效率低、維護成本高、權限管理複雜。系統需要支援：

1. **完整的權限管理體系：** 支援使用者、角色、組織的多層級權限管理，並提供組織圖視覺化
2. **檔案管理功能：** 提供檔案上傳、下載、分享、預覽等功能，支援多使用者隔離
3. **系統監控與維護：** 提供硬體監控（CPU、GPU、記憶體、網路、儲存、電源）、日誌管理、快取清理等功能
4. **AI 功能整合：** 整合物件辨識（ONNX 模型）、圖像生成（DALL-E 3）、向量資料庫（Qdrant）等 AI 功能
5. **彈性的導航管理：** 支援動態側邊欄導航，可根據使用者權限顯示不同的導航項目

此外，系統需要提供良好的擴展性與維護性，支援多專案管理、團隊協作等功能。

**系統定位：**

本系統在整體架構中屬於企業級後台管理系統，作為其他系統的管理入口與基礎平台。系統的核心定位是：

1. **權限管理平台：** 統一管理所有系統的使用者、角色、組織與權限設定
2. **系統管理中心：** 提供系統監控、日誌管理、檔案管理等基礎功能
3. **AI 功能平台：** 整合各種 AI 功能（物件辨識、圖像生成、向量搜尋），提供統一的 AI 服務
4. **擴展平台：** 支援多專案管理、團隊協作等功能，為未來擴展提供基礎

系統採用 Areas 架構，主要分為 Backoffice（後台管理）與 NovaAPP（應用程式）兩個區域，目前以 Backoffice 為主。

**技術選型的思考過程：**

#### ASP.NET Core 8.0 的選型分析

**問題識別：**

需要建立一個現代化的企業級後台管理系統，支援完整的權限管理、檔案管理、系統監控、AI 功能整合等。系統需要良好的擴展性、效能與維護性，並能夠整合各種外部服務（AI API、向量資料庫等）。

**方案評估：**

| 方案 | 優點 | 缺點 | 適用場景 | 不選擇的原因 |
|------|------|------|----------|-------------|
| ASP.NET Core 8.0 | • 現代化架構<br>• 跨平台支援<br>• 效能優異<br>• 長期支援<br>• 豐富的擴展生態 | • 需要學習新框架<br>• 設定較複雜 | 新專案、需要現代化架構 | - |
| ASP.NET MVC Framework | • 穩定成熟<br>• 團隊熟悉<br>• 設定簡單 | • 較舊的技術<br>• 只能在 Windows 運行<br>• 效能較差<br>• 長期支援有限 | 既有系統維護、Windows 限定環境 | 專案需要跨平台支援，且需要現代化的功能與效能 |
| Node.js / Express | • 非同步效能佳<br>• JavaScript 全端<br>• 生態豐富 | • 型別安全性較差<br>• 團隊熟悉度 | 前後端統一技術棧 | 團隊對 .NET 熟悉，且需要強型別支援 |

**決策依據：**

本專案是一個新建立的企業級後台管理系統，需要現代化的架構與良好的擴展性。ASP.NET Core 8.0 提供了：

1. **現代化架構：** 依賴注入、中介軟體、過濾器等現代化特性
2. **跨平台支援：** 可以在 Windows、Linux、macOS 運行，提升部署彈性
3. **效能優異：** 相比 ASP.NET MVC Framework 有顯著的效能提升
4. **豐富的擴展：** 可以輕鬆整合 AI 服務、向量資料庫等外部服務
5. **長期支援：** .NET 8 是長期支援版本，提供長期維護保證

**實際應用與調整：**

實際應用中，使用 Areas 架構劃分不同的功能區域（Backoffice、NovaAPP），每個 Area 有獨立的 Controllers、Views、Services、Repositories。使用依賴注入統一管理服務，使用中介軟體處理通用邏輯（錯誤處理、IP 安全檢查、硬體監控等）。這樣的設計提供了清晰的架構邊界，易於維護與擴展。

**可複用的思考模式：**

當選擇後端框架時，應該考慮：1. 專案的長期維護需求 2. 團隊的技術熟悉度 3. 系統的效能要求 4. 擴展性與整合需求。選擇現代化框架雖然有學習成本，但可以提升長期競爭力與維護性。

#### Entity Framework Core 8.0 與 PostgreSQL 的選型分析

**問題識別：**

系統需要存取兩個不同的資料庫：Hnbdata（專案資料、團隊成員、存取記錄等）與 HnbHnbBackoffice（後台管理相關資料，如權限、導航、檔案管理等）。每個資料庫有獨立的業務領域，資料模型獨立。系統需要選擇資料庫類型與 ORM 框架。

**方案評估：**

| 方案 | 優點 | 缺點 | 適用場景 | 不選擇的原因 |
|------|------|------|----------|-------------|
| EF Core 8.0 + PostgreSQL | • 強型別支援<br>• LINQ 查詢<br>• 遷移管理<br>• 跨平台<br>• 開源免費<br>• 效能優異 | • 學習成本較高<br>• 設定較複雜 | 現代化應用、需要 ORM 功能 | - |
| EF Core + SQL Server | • 強型別支援<br>• LINQ 查詢<br>• 與 Windows 整合良好<br>• 團隊熟悉 | • 授權成本高<br>• 跨平台支援較差 | Windows 限定環境、已有 SQL Server | 專案需要跨平台支援，且希望降低授權成本 |
| Dapper + PostgreSQL | • 效能較佳<br>• 靈活性高<br>• 學習曲線低 | • 需要手寫 SQL<br>• 型別安全性較低<br>• 無遷移功能 | 效能要求極高的場景 | 本專案需要型別安全與 LINQ 查詢，效能要求不極端 |
| Npgsql（直接使用） | • 效能最佳<br>• 完全控制 | • 需要大量手寫程式碼<br>• 維護成本高<br>• 容易出錯 | 極端效能要求的場景 | 開發效率與維護成本不划算 |

**決策依據：**

本專案選擇 PostgreSQL 作為資料庫，EF Core 8.0 作為 ORM 框架，主要考慮：

1. **跨平台支援：** PostgreSQL 可以在多種作業系統運行，與 ASP.NET Core 的跨平台特性匹配
2. **開源免費：** PostgreSQL 是開源資料庫，不需要授權成本，適合企業內部使用
3. **效能優異：** PostgreSQL 在大量資料處理與複雜查詢方面表現優異
4. **功能豐富：** PostgreSQL 支援 JSON、陣列、全文搜尋等進階功能，適合複雜的資料需求
5. **EF Core 整合：** EF Core 8.0 對 PostgreSQL 有良好的支援，提供強型別與 LINQ 查詢

使用 EF Core 而非 Dapper 的原因是：本專案需要強型別支援、LINQ 查詢、遷移管理等功能，可以大幅提升開發效率與程式碼品質。雖然效能略遜於 Dapper，但在可接受範圍內。

**實際應用與調整：**

實際應用中，每個資料庫使用獨立的 DbContext（HnbdataDbContext、HnbHnbBackofficeDbContext），在 `Program.cs` 中分別註冊。連線字串在 `appsettings.json` 中設定，使用 Npgsql 連線 PostgreSQL。這樣的設計確保了資料隔離，每個資料庫的資料模型互不干擾。

**可複用的思考模式：**

當選擇資料庫與 ORM 時，應該考慮：1. 跨平台需求 2. 授權成本 3. 效能要求 4. 團隊熟悉度 5. 功能需求。選擇開源資料庫可以降低授權成本，選擇現代化的 ORM 可以提升開發效率。

#### Areas 架構的選型分析

**問題識別：**

系統需要支援多個功能區域：Backoffice（後台管理）、NovaAPP（應用程式，未來可能擴展）。每個區域有獨立的業務邏輯、頁面與權限控制。需要一個清晰的架構來組織這些區域。

**方案評估：**

| 方案 | 優點 | 缺點 | 適用場景 | 不選擇的原因 |
|------|------|------|----------|-------------|
| Areas 架構 | • 清晰的功能邊界<br>• 易於組織大型專案<br>• 支援獨立的路由與權限控制<br>• ASP.NET Core 原生支援 | • 設定較複雜<br>• 需要理解 Areas 概念 | 大型專案、需要功能分區 | - |
| 單一 Controller/View 結構 | • 設定簡單<br>• 結構清晰 | • 不適合大型專案<br>• 難以管理多個功能區域 | 小型專案、單一功能區域 | 專案規模較大，且有明確的功能分區需求 |
| 微服務架構 | • 完全獨立<br>• 可獨立部署<br>• 技術棧可不同 | • 複雜度高<br>• 維護成本高<br>• 需要服務間通訊 | 大型分散式系統 | 專案規模尚未達到微服務的需求，Areas 架構已足夠 |

**決策依據：**

本專案採用 Areas 架構，主要考慮：

1. **功能邊界清晰：** Backoffice 與 NovaAPP 有不同的業務邏輯與使用者群體，需要清晰的邊界
2. **易於擴展：** 未來可以輕鬆新增新的 Area（如 Mobile、API 等），不影響既有功能
3. **獨立的路由與權限：** 每個 Area 可以有獨立的路由規則與權限控制
4. **符合 ASP.NET Core 最佳實踐：** Areas 是 ASP.NET Core 推薦的大型專案組織方式

雖然設定較複雜，但提供了清晰的架構邊界，長期來看可以提升可維護性。

**實際應用與調整：**

實際應用中，每個 Area 有獨立的 Controllers、Views、Services、Repositories。使用 BaseController 提供共用的功能（如側邊欄載入），使用 PermissionAttribute 提供統一的權限控制。路由設定在 `Program.cs` 中使用 `MapControllerRoute` 設定，支援 Areas 路由。

**可複用的思考模式：**

當組織大型專案時，應該考慮：1. 功能邊界的清晰度 2. 未來的擴展需求 3. 維護的便利性。Areas 架構可以提供清晰的功能邊界，但不應該過度使用，只有在有明確功能分區需求時才使用。

#### AI 功能整合的選型分析

**問題識別：**

系統需要整合多種 AI 功能：物件辨識（ONNX 模型）、圖像生成（DALL-E 3）、向量搜尋（Qdrant）。這些功能需要與系統的其他功能（如檔案管理、權限控制）整合。

**方案評估：**

| 方案 | 優點 | 缺點 | 適用場景 | 不選擇的原因 |
|------|------|------|----------|-------------|
| ONNX Runtime + DALL-E 3 API + Qdrant | • ONNX 模型可在本地運行<br>• DALL-E 3 API 提供高品質圖像生成<br>• Qdrant 提供高效能向量搜尋<br>• 各自獨立，可分別擴展 | • 需要整合多個系統<br>• 設定複雜 | 需要多種 AI 功能的場景 | - |
| 單一 AI 平台（如 Azure Cognitive Services） | • 統一管理<br>• 設定簡單<br>• 支援多種 AI 功能 | • 授權成本高<br>• 依賴特定平台<br>• 功能可能受限 | 預算充足、依賴特定平台 | 專案需要更靈活的整合方式，且希望降低授權成本 |
| 自行開發 AI 功能 | • 完全控制<br>• 無授權成本 | • 開發成本高<br>• 技術難度高<br>• 維護成本高 | 有 AI 專業團隊 | 專案不需要從零開發，使用現有工具更有效率 |

**決策依據：**

本專案選擇整合多個 AI 系統，主要考慮：

1. **功能需求多樣化：** 需要物件辨識、圖像生成、向量搜尋等多種功能，單一平台可能無法滿足所有需求
2. **成本考量：** ONNX 模型可以在本地運行，降低 API 呼叫成本。DALL-E 3 API 按使用量計費，成本可控
3. **技術靈活性：** 使用開源工具（ONNX Runtime、Qdrant）可以避免廠商鎖定，未來可以替換其他工具
4. **效能考量：** ONNX 模型在本地運行可以提供低延遲的辨識服務，適合即時應用

雖然需要整合多個系統，但提供了更好的功能選擇與成本控制。

**實際應用與調整：**

實際應用中，將 AI 功能封裝為獨立模組（IntelligentSystems），每個模組有自己的配置、API 與服務。使用 ModelHealthChecker 管理 ONNX 模型的健康狀態，自動下載與初始化模型。使用 Dependency Injection 注入 AI 模組到需要使用的服務中。

**可複用的思考模式：**

當整合多個外部服務時，應該考慮：1. 功能需求的完整性 2. 成本效益 3. 技術靈活性 4. 維護成本。選擇開源工具可以降低授權成本，但需要考慮維護成本。封裝為獨立模組可以提升可維護性與可測試性。

**整體架構設計的演進：**

**初始設計思路：**

最初設計時，考慮將所有功能放在單一的 Controller/View 結構中，這樣的設計雖然簡單，但隨著功能增加，會導致結構混亂，難以維護。

**遇到的問題與調整：**

- **問題一：** 功能模組過多，Controller 與 View 結構混亂
  - **問題的根本原因：** 沒有明確的功能邊界，所有功能混在一起
  - **第一版的不足：** 可維護性差，難以擴展，功能間耦合度高

- **調整方案：**
  - **調整思路：** 採用 Areas 架構，將功能劃分為不同的區域（Backoffice、NovaAPP）
  - **方案比較：** 考慮過使用多專案架構，但 Areas 架構提供了更好的整合性，且可以共用基礎設施（認證、授權等）
  - **改進後的優勢：** 每個 Area 有清晰的邊界，易於維護與擴展。使用 BaseController 提供共用功能，降低重複程式碼

**當前架構的設計原則：**

1. **功能分區原則：** 使用 Areas 架構劃分不同的功能區域，每個 Area 有獨立的業務邏輯
2. **依賴注入原則：** 所有服務透過 Dependency Injection 注入，降低耦合度
3. **分層架構原則：** Controller → Service → Repository 三層架構，明確職責分離
4. **模組化原則：** AI 功能封裝為獨立模組（IntelligentSystems），可以獨立測試與維護

**未來可能的改進方向：**

1. 考慮將 AI 模組獨立為微服務，提供更好的擴展性與效能
2. 引入 CQRS 模式，分離讀寫操作，提升查詢效能
3. 考慮使用 GraphQL 提供更靈活的 API 查詢介面

## 3. 環境設定

### 3.1 開發環境設定

**本專案的特定設定考量：**

本專案使用 ASP.NET Core 8.0，需要在 Visual Studio 2022 中開啟專案。專案使用 `appsettings.json` 與 `appsettings.Development.json` 管理設定，包括資料庫連線字串、AI 服務設定、JWT 設定等。

專案使用依賴注入管理服務，所有服務在 `Extensions/ServiceExtensions.cs` 中註冊。專案使用中介軟體處理通用邏輯（錯誤處理、IP 安全檢查、硬體監控等）。

### 3.2 資料庫/外部服務連線設定

3. **模組化原則：** AI 功能、工具類別、中介軟體等都以模組方式組織，提升可重用性
4. **Partial View 組織：** Views 使用 Partials 組織，每個頁面有自己的 Partials 目錄，Modal 放在 Modal 子目錄中

**重要檔案的特殊用途：**

1. **Program.cs：** 應用程式的啟動與設定入口，註冊所有服務、中介軟體、路由等
2. **ServiceExtensions.cs：** 統一管理所有依賴注入設定，包含 Services、Repositories、Utilities、AI 模組等
3. **BaseController：** 提供所有後台管理控制器的共用功能（如側邊欄載入），並套用權限控制
4. **ModelHealthChecker：** 管理 ONNX 模型的健康狀態，自動下載與初始化模型，確保 AI 功能可用

2. **HnbHnbBackoffice：** 儲存後台管理相關的資料（權限、導航、檔案管理、硬體監控、AI 配置等）

   - 利用檔案系統的儲存效能（大檔案不佔用資料庫空間）
   - 支援檔案分享與權限控制（透過資料庫記錄擁有者與分享對象）

3. **AI 服務與資料庫：** AI 配置（ai_config）與 AI 日誌（ai_log）儲存在資料庫，但 AI 模型檔案（ONNX）儲存在檔案系統。AI 服務透過 HTTP 或 SDK 連線到外部服務（DALL-E 3 API、Qdrant），不直接操作資料庫。

1. **組織階層關係：** permission_management 使用 parent_id 建立組織階層關係，可以建立多層級的組織架構。查詢組織時需要遞迴查詢，或使用檢視表簡化查詢。

2. **檔案擁有者關係：** file_manager 透過 owner_id 關聯到 permission_management（使用者），建立檔案擁有者關係。檔案分享功能需要額外的資料表或欄位來記錄分享對象。

3. **跨資料庫關聯：** team_member 與 permission_management 分屬不同資料庫，關聯關係在應用程式層面處理。查詢團隊成員時需要同時查詢兩個資料庫，在應用程式層面組合資料。

### 5.3 關鍵資料表設計說明

#### HnbHnbBackoffice.permission_management

**本專案中此資料表的設計目的：**

permission_management 資料表是權限管理的核心資料表，統一儲存使用者、角色、組織的資料。使用 `type` 欄位區分不同的實體類型（使用者、角色、組織），這樣的設計可以統一管理權限，簡化權限控制的邏輯。

**專案特定的設計考量：**

- **type 欄位：** 使用字串區分實體類型（如 "user"、"role"、"organization"），統一管理不同類型的權限實體
- **parent_id 欄位：** 用於建立組織階層關係，組織可以有上層組織，形成樹狀結構
- **統一權限模型：** 使用者、角色、組織都使用相同的資料表，簡化權限管理邏輯
- **檢視表輔助：** 使用 `vw_permission_user`、`vw_permission_role`、`vw_permission_organization` 等檢視表提供特定類型的查詢介面

**本專案中資料的生命週期：**

1. **新增：** 透過權限管理頁面新增使用者、角色或組織，系統驗證必填欄位與唯一性
2. **修改：** 透過權限管理頁面修改資料，系統驗證後更新
3. **刪除：** 使用邏輯刪除或實體刪除（依業務規則而定），刪除前需要檢查是否有其他資料關聯
4. **查詢：** 透過檢視表查詢特定類型的資料，支援權限範圍過濾（OrganizationScope）

**專案中的使用情境：**

- **使用者管理：** 查詢、新增、修改、刪除使用者，設定使用者的角色與組織歸屬
- **角色管理：** 查詢、新增、修改、刪除角色，設定角色的導航權限
- **組織管理：** 查詢、新增、修改、刪除組織，建立組織階層關係，查詢組織圖

#### HnbHnbBackoffice.file_manager

**本專案中此資料表的設計目的：**

file_manager 資料表儲存檔案管理的元資料，包括檔案名稱、路徑、大小、擁有者、分享對象等。實際檔案儲存在檔案系統，資料庫只儲存元資料，這樣的設計可以：

1. **利用資料庫的查詢能力：** 透過 SQL 查詢檔案、排序、篩選
2. **支援檔案分享與權限控制：** 透過資料庫記錄擁有者與分享對象，實現檔案分享功能
3. **避免資料庫儲存大檔案：** 實際檔案儲存在檔案系統，不佔用資料庫空間

**專案特定的設計考量：**

- **virtual_path 欄位：** 虛擬路徑，用於使用者介面顯示，與實際檔案路徑可能不同
- **actual_path 欄位：** 實際檔案路徑，對應檔案系統中的路徑
- **owner_id 欄位：** 檔案擁有者（使用者 ID），用於權限控制
- **shared_with 欄位：** 分享對象（可能為 JSON 或關聯資料表），記錄可以存取此檔案的使用者

**本專案中資料的生命週期：**

1. **新增：** 使用者上傳檔案時，系統儲存實際檔案到檔案系統，並在資料庫中建立元資料記錄
2. **修改：** 修改檔案名稱、路徑、分享對象等元資料
3. **刪除：** 執行邏輯刪除或實體刪除（依業務規則而定），實際檔案可能立即刪除或延後刪除
4. **查詢：** 根據使用者權限查詢檔案，支援路徑篩選、檔案類型篩選等

**專案中的使用情境：**

- **檔案上傳：** 使用者上傳檔案，系統儲存到檔案系統並建立元資料
- **檔案查詢：** 根據路徑查詢檔案清單，顯示檔案資訊（名稱、大小、修改時間等）
- **檔案分享：** 設定檔案的分享對象，允許其他使用者存取
- **檔案下載：** 根據權限檢查後提供檔案下載

#### HnbHnbBackoffice.hardware_monitoring

**本專案中此資料表的設計目的：**

hardware_monitoring 資料表儲存硬體監控資料，包括 CPU、GPU、記憶體、網路、儲存、電源等監控資訊。這些資料由 HardwareMonitoringMiddleware 定期收集並寫入資料庫，用於系統健康狀態監控。

**專案特定的設計考量：**

- **is_active 欄位：** 監控服務是否啟用，用於控制監控功能的開關
- **各種監控欄位：** cpu_usage_percent、memory_usage_percent、gpu_usage_percent 等，儲存各種硬體資源的使用率
- **監控時間：** 記錄監控資料的收集時間，用於分析趨勢

**本專案中資料的生命週期：**

1. **自動收集：** HardwareMonitoringMiddleware 定期收集硬體監控資料並寫入資料庫
2. **查詢顯示：** 透過 Dashboard 或 Settings 頁面查詢並顯示最新的監控資料
3. **歷史資料：** 保留歷史監控資料，用於分析趨勢與問題診斷

**專案中的使用情境：**

- **系統狀態監控：** Dashboard 顯示系統的整體健康狀態（CPU、記憶體使用率等）
- **硬體監控管理：** Settings 頁面提供硬體監控的詳細資訊與設定
- **問題診斷：** 透過歷史監控資料分析系統效能問題

### 5.4 資料庫直接操作說明

1. **連線資訊：** 參考 3.2 節連線設定
2. **資料表關聯關係：** 參考 5.2 節
3. **注意事項：**
   - 手動修正資料前建議先備份
   - 修改 permission_management 資料時，需要確認組織階層關係的正確性
   - 刪除檔案記錄時，需要確認實際檔案是否也需要刪除
   - 修改權限設定時，需要確認對應的使用者、角色、組織關係

**本專案中需要注意這些事項的原因：**

1. **組織階層關係的重要性：** permission_management 使用 parent_id 建立組織階層關係，如果階層關係錯誤，會導致組織圖顯示錯誤、權限範圍計算錯誤等問題。修改組織資料時需要特別注意階層關係。

2. **檔案元資料與實際檔案的一致性：** file_manager 只儲存檔案元資料，實際檔案儲存在檔案系統。如果只刪除資料庫記錄而不刪除實際檔案，會造成檔案系統空間浪費。如果只刪除實際檔案而不刪除資料庫記錄，會導致查詢到不存在的檔案。

3. **權限關係的完整性：** permission_management 統一管理使用者、角色、組織，修改任何一個實體時都需要確認對應的關係（如使用者的角色歸屬、角色的導航權限等）是否正確。

### 6.2 各模組詳細說明

#### Dashboard（儀表板）模組

**本專案的需求背景：**

使用者登入後需要一個總覽頁面，快速了解系統狀態、存取常用功能、查看近期活動等。傳統的分散式頁面設計讓使用者難以快速掌握系統整體狀況。

**主要功能：**

- **功能一：** 系統狀態顯示 - 顯示系統的整體健康狀態（伺服器狀態、CPU 使用率、記憶體使用率等）
- **功能二：** 快速操作 - 提供常用功能的快速存取連結
- **功能三：** 近期活動顯示 - 顯示最近的系統活動或專案資訊
- **功能四：** 歡迎訊息 - 顯示使用者的歡迎訊息與基本資訊

**查詢資料來源：**

- HnbHnbBackoffice.hardware_monitoring：硬體監控資料（CPU、記憶體使用率等）
- Hnbdata.project：專案資料（用於顯示近期活動）

**專案中這些資料的用途：**

- hardware_monitoring：提供系統狀態的即時資訊，幫助使用者快速了解系統健康狀態
- project：顯示專案列表或最近的專案活動，幫助使用者快速了解專案狀況

**異動資料來源：**

無（此模組只提供查詢功能，不負責新增或修改資料）

**本專案的操作邏輯：**

1. **頁面載入：** 使用者登入後進入 Dashboard，系統載入最新的硬體監控資料與專案資料，顯示在頁面上
2. **即時更新：** 系統狀態可能需要定期更新（透過 AJAX），但本模組目前可能只顯示載入時的資料

**專案特定的規則：**

- **資料載入時機：** Dashboard 頁面載入時才查詢硬體監控資料，不提供即時自動更新。如果需要即時更新，需要在前端使用定時器呼叫 AJAX 更新。

**規則原因：**

Dashboard 是總覽頁面，主要目的是快速了解系統狀態，不需要即時更新。如果使用者需要即時的監控資料，可以前往 Settings 頁面查看詳細的硬體監控資訊。

**本專案中模組間的關聯：**

Dashboard 模組依賴 Settings 模組提供硬體監控資料，依賴 PermissionManagement 模組提供使用者資訊。Dashboard 的快速操作連結可能指向其他模組（如 FileManager、PermissionManagement 等）。

#### PermissionManagement（權限管理）模組

**本專案的需求背景：**

企業需要一個完整的權限管理系統來管理使用者、角色、組織與權限。傳統的簡單權限控制無法滿足複雜的組織架構與權限需求。系統需要支援：

1. **多層級組織架構：** 支援組織階層關係，可以建立多層級的組織架構
2. **角色權限管理：** 支援角色的導航權限設定，可以控制角色可以存取哪些功能
3. **使用者角色歸屬：** 使用者可以屬於多個角色，角色可以屬於多個組織
4. **組織圖視覺化：** 提供組織圖的視覺化顯示，幫助管理者了解組織架構

**主要功能：**

- **功能一：** 使用者管理 - 查詢、新增、修改、刪除使用者，設定使用者的角色與組織歸屬
- **功能二：** 角色管理 - 查詢、新增、修改、刪除角色，設定角色的導航權限
- **功能三：** 組織管理 - 查詢、新增、修改、刪除組織，建立組織階層關係
- **功能四：** 組織圖顯示 - 視覺化顯示組織架構，支援展開/收合、拖曳等功能

**查詢資料來源：**

- HnbHnbBackoffice.permission_management：權限管理主檔
- HnbHnbBackoffice.vw_permission_user：使用者權限檢視表
- HnbHnbBackoffice.vw_permission_role：角色權限檢視表
- HnbHnbBackoffice.vw_permission_organization：組織權限檢視表

**專案中這些資料的用途：**

- permission_management：統一儲存使用者、角色、組織的資料與權限
- vw_permission_user：提供使用者的整合查詢介面，包含使用者的組織資訊
- vw_permission_role：提供角色的整合查詢介面，包含角色的權限資訊
- vw_permission_organization：提供組織的整合查詢介面，包含組織的成員資訊

**異動資料來源：**

- HnbHnbBackoffice.permission_management：新增、修改、刪除使用者、角色、組織的資料

**本專案的操作邏輯：**

1. **使用者管理：**
   - 查詢：根據組織範圍（OrganizationScope）過濾使用者清單，只顯示當前使用者有權限查看的組織的使用者
   - 新增/修改：驗證必填欄位（使用者名稱、密碼、組織等），設定使用者的角色歸屬（多對多關係）
   - 刪除：檢查使用者是否有其他資料關聯（如檔案擁有者、團隊成員等），確認後刪除

2. **角色管理：**
   - 查詢：根據組織範圍過濾角色清單
   - 新增/修改：驗證必填欄位，設定角色的導航權限（可能為字串陣列或 JSON）
   - 刪除：檢查角色是否有使用者關聯，確認後刪除

3. **組織管理：**
   - 查詢：根據組織範圍過濾組織清單，支援組織階層關係的查詢
   - 新增/修改：驗證必填欄位，設定組織的上層組織（parent_id），建立組織階層關係
   - 刪除：檢查組織是否有下層組織或成員，確認後刪除

4. **組織圖顯示：**
   - 查詢：遞迴查詢組織階層關係，建立樹狀結構
   - 顯示：使用視覺化工具（如 Canvas、D3.js 等）顯示組織圖，支援展開/收合、拖曳等功能

**專案特定的規則：**

- **組織範圍（OrganizationScope）：** 使用者只能管理自己組織範圍內的使用者、角色、組織。這是權限控制的核心規則，確保不同組織的資料隔離。

**規則原因：**

企業通常有多個組織（部門、分公司等），不同組織的資料應該隔離。OrganizationScope 確保使用者只能管理自己組織範圍內的資料，提升資料安全性。

**本專案中模組間的關聯：**

PermissionManagement 模組是系統的核心模組，其他模組都依賴此模組進行權限控制：

- **Authorize 模組：** 使用 PermissionManagement 驗證使用者身份
- **BaseController：** 使用 PermissionAttribute 檢查使用者權限
- **SidebarNavigation 模組：** 根據使用者的角色權限顯示不同的導航項目
- **FileManager 模組：** 使用 PermissionManagement 檢查檔案的存取權限

#### FileManager（檔案管理）模組

**本專案的需求背景：**

企業需要一個檔案管理系統來管理檔案的上傳、下載、分享、預覽等。傳統的檔案伺服器方式缺乏權限控制與管理功能。系統需要支援：

1. **多使用者檔案隔離：** 每個使用者有自己的檔案儲存空間，檔案之間隔離
2. **檔案分享功能：** 使用者可以將檔案分享給其他使用者
3. **檔案預覽功能：** 支援多種檔案類型的預覽（圖片、PDF、文字檔等）
4. **資料夾管理：** 支援資料夾的新增、刪除、重新命名等

**主要功能：**

- **功能一：** 檔案上傳 - 支援單檔案與多檔案上傳，支援資料夾結構上傳
- **功能二：** 檔案查詢 - 根據路徑查詢檔案清單，支援多種檢視模式（清單、網格等）
- **功能三：** 檔案下載 - 根據權限檢查後提供檔案下載
- **功能四：** 檔案預覽 - 支援圖片、PDF、文字檔等類型的預覽
- **功能五：** 檔案分享 - 設定檔案的分享對象，允許其他使用者存取
- **功能六：** 資料夾管理 - 新增、刪除、重新命名資料夾

**查詢資料來源：**

- HnbHnbBackoffice.file_manager、vw_file_manager：檔案元資料
- 檔案系統（Areas/Backoffice/storage）：實際檔案

**專案中這些資料的用途：**

- file_manager：儲存檔案的元資料（名稱、路徑、大小、擁有者等），用於查詢、排序、權限控制
- 檔案系統：儲存實際檔案，提供檔案存取

**異動資料來源：**

- HnbHnbBackoffice.file_manager：新增、修改、刪除檔案元資料
- 檔案系統：新增、刪除實際檔案

**本專案的操作邏輯：**

1. **檔案上傳：**
   - 接收上傳的檔案，驗證檔案大小與類型
   - 儲存實際檔案到使用者的儲存目錄（Areas/Backoffice/storage/{username}/...）
   - 建立檔案元資料記錄到 file_manager，設定擁有者為當前使用者
   - 支援資料夾結構上傳，自動建立對應的資料夾

2. **檔案查詢：**
   - 根據路徑查詢檔案清單，過濾當前使用者有權限查看的檔案（擁有者或分享對象）
   - 支援多種檢視模式（清單、網格、詳細資料等）
   - 支援檔案類型篩選、排序等功能

3. **檔案下載：**
   - 檢查使用者是否有權限存取此檔案（擁有者或分享對象）
   - 如果無權限，返回 403 Forbidden
   - 如果有權限，從檔案系統讀取檔案並返回給使用者

4. **檔案分享：**
   - 檢查使用者是否為檔案擁有者
   - 設定檔案的分享對象（shared_with 欄位），允許其他使用者存取
   - 更新檔案元資料

5. **資料夾管理：**
   - 新增資料夾：在檔案系統中建立資料夾，並在資料庫中建立對應的記錄
   - 刪除資料夾：刪除資料夾及其所有內容（遞迴刪除），並刪除資料庫記錄
   - 重新命名資料夾：更新檔案系統中的資料夾名稱與資料庫記錄

**專案特定的規則：**

- **使用者儲存隔離：** 每個使用者有自己的儲存目錄（Areas/Backoffice/storage/{username}/），檔案之間隔離。這是為了確保不同使用者的檔案不會互相干擾。
- **檔案分享權限：** 只有檔案擁有者可以設定檔案的分享對象。分享對象可以存取檔案，但無法修改或刪除檔案（除非擁有者明確授權）。
- **虛擬路徑與實際路徑：** 使用者看到的是虛擬路徑（如 `/MyFiles/...`），實際檔案儲存在使用者的儲存目錄中。這樣的設計可以隱藏實際檔案系統結構，提供更友善的使用者介面。

**規則原因：**

虛擬路徑提供更友善的使用者體驗，使用者不需要了解實際檔案系統結構。實際路徑隱藏在系統內部，提升安全性。

**本專案中模組間的關聯：**

FileManager 模組依賴 PermissionManagement 模組進行權限控制，檢查檔案的存取權限。

#### Settings（系統設定）模組

**本專案的需求背景：**

系統需要提供系統監控、日誌管理、快取清理、維護模式等功能，協助管理者監控系統狀態、維護系統運作。傳統的分散式工具無法提供統一的系統管理介面。

**主要功能：**

- **功能一：** 系統狀態監控 - 顯示系統的整體健康狀態（CPU、記憶體、GPU、網路、儲存、電源等）
- **功能二：** 日誌管理 - 查詢、匯出、清理錯誤日誌與存取記錄
- **功能三：** 快取清理 - 清理記憶體快取，釋放系統資源
- **功能四：** 維護模式 - 開啟/關閉維護模式，控制系統存取
- **功能五：** 硬體監控 - 顯示硬體監控的詳細資訊與歷史資料

**查詢資料來源：**

- HnbHnbBackoffice.hardware_monitoring、vw_hardware_monitoring：硬體監控資料
- Hnbdata.error_logs：錯誤日誌
- Hnbdata.access_records：存取記錄
- 快取系統：記憶體快取資料

**專案中這些資料的用途：**

- hardware_monitoring：提供系統狀態的即時資訊，幫助管理者監控系統健康狀態
- error_logs：記錄系統錯誤與例外，用於問題診斷與系統維護
- access_records：記錄使用者的存取行為，用於安全審計

**異動資料來源：**

- HnbHnbBackoffice.hardware_monitoring：硬體監控資料由 HardwareMonitoringMiddleware 自動更新
- Hnbdata.error_logs：錯誤日誌由 ExceptionLoggingMiddleware 自動記錄，管理者可以手動清理
- 快取系統：管理者可以手動清理快取

**本專案的操作邏輯：**

1. **系統狀態監控：**
   - 查詢最新的硬體監控資料，顯示 CPU、記憶體、GPU、網路、儲存、電源等使用率
   - 支援歷史資料查詢，分析趨勢與問題

2. **日誌管理：**
   - 查詢錯誤日誌與存取記錄，支援多條件篩選（時間、類型、使用者等）
   - 匯出日誌為檔案，用於離線分析
   - 清理舊日誌（可選擇清理30天前的日誌），釋放資料庫空間

3. **快取清理：**
   - 清理所有快取或特定類型的快取
   - 可選擇清理30天前的快取資料

4. **維護模式：**
   - 開啟維護模式時，系統拒絕所有非管理者的存取
   - 關閉維護模式時，恢復正常運作

5. **硬體監控：**
   - 顯示硬體監控的詳細資訊，包括 CPU、GPU、記憶體、網路、儲存、電源等
   - 支援歷史資料查詢與趨勢分析

**專案特定的規則：**

- **硬體監控自動化：** HardwareMonitoringMiddleware 定期收集硬體監控資料並寫入資料庫，無需手動觸發。這是為了確保監控資料的即時性與完整性。

**規則原因：**

硬體監控需要持續收集資料，才能及時發現系統問題。自動化的監控機制可以確保資料的完整性與即時性。

**本專案中模組間的關聯：**

Settings 模組使用 HardwareMonitoringMiddleware 收集硬體監控資料，使用 ExceptionLoggingMiddleware 記錄錯誤日誌。Settings 模組提供系統監控與維護功能，其他模組的錯誤日誌可能在此模組中查看。

- **功能一：** 導航項目管理 - 查詢、新增、修改、刪除導航項目

**異動資料來源：**

- HnbHnbBackoffice.sidebar_navigation：新增、修改、刪除導航項目

**本專案的操作邏輯：**

1. **導航項目管理：**
   - 查詢：根據條件查詢導航項目清單

**本專案中模組間的關聯：**

Database 模組使用 DynamicDbContext 動態建立資料庫連線。Database 模組提供資料庫管理功能，可能與其他模組共用資料庫連線功能。

## 7. 重要功能詳解

### 7.1 權限管理系統

#### 7.1.1 功能設計背景

**需要解決的問題：**

企業需要一個完整的權限管理系統來管理使用者、角色、組織與權限。傳統的簡單權限控制無法滿足複雜的組織架構與權限需求。系統需要支援多層級組織架構、角色權限管理、使用者角色歸屬、組織圖視覺化等功能。

**問題的複雜度：**

權限管理涉及多個面向：

1. **多層級組織架構：** 支援組織階層關係，可以建立多層級的組織架構，查詢時需要遞迴查詢或使用檢視表
2. **角色權限管理：** 支援角色的導航權限設定，可以控制角色可以存取哪些功能，權限以字串陣列或 JSON 儲存
3. **使用者角色歸屬：** 使用者可以屬於多個角色，角色可以屬於多個組織，這是多對多的關係
4. **組織範圍控制：** 使用者只能管理自己組織範圍內的資料，需要實現組織範圍過濾（OrganizationScope）

#### 7.1.2 設計決策的思考過程

**設計目標：**

建立一個統一且靈活的權限管理系統，可以支援複雜的組織架構與權限需求。系統應該提供清晰的權限控制邏輯，易於維護與擴展。

**設計方案比較：**

| 方案 | 優點 | 缺點 | 適用場景 | 為什麼不採用 |
|------|------|------|----------|-------------|
| 統一資料表（permission_management） | • 統一管理<br>• 邏輯簡單<br>• 易於維護 | • 資料表結構較複雜<br>• 需要 type 欄位區分 | 需要統一管理不同類型的權限實體 | - |
| 分離資料表（users、roles、organizations） | • 結構清晰<br>• 符合正規化原則 | • 邏輯複雜<br>• 維護成本高<br>• 權限控制困難 | 不同類型實體差異較大 | 本專案需要統一管理權限，分離資料表會增加複雜度 |

**最終設計的考量：**

| 設計決策點 | 選擇的方案 | 為什麼這樣選擇 | 犧牲了什麼 | 如何權衡 |
|-----------|----------|--------------|-----------|----------|
| 資料表設計 | 統一使用 permission_management，使用 type 欄位區分 | 統一管理使用者、角色、組織，簡化權限控制邏輯 | 資料表結構較複雜，需要 type 欄位判斷 | 統一管理可以簡化權限控制，提升維護性 |
| 組織階層關係 | 使用 parent_id 建立自我關聯 | 支援多層級組織架構，查詢時可以使用遞迴或檢視表 | 查詢需要遞迴，效能可能受影響 | 組織階層關係對業務很重要，可以透過檢視表優化查詢 |
| 角色權限儲存 | 使用字串陣列或 JSON 儲存導航權限 | 提供靈活的權限設定，支援動態權限調整 | 無法使用資料庫外鍵約束，需要在應用程式層面驗證 | 靈活性比外鍵約束更重要，可以在應用程式層面驗證 |
| 組織範圍控制 | 使用 OrganizationScope 過濾資料 | 確保使用者只能管理自己組織範圍內的資料，提升資料安全性 | 查詢時需要額外過濾，可能影響效能 | 資料安全性比效能更重要，可以透過索引優化查詢 |

**從這個設計中學到的模式：**

當設計權限管理系統時，應該考慮：1. 權限實體的統一性與分離性的權衡 2. 組織階層關係的儲存與查詢方式 3. 權限的儲存方式（外鍵關聯 vs. 字串陣列）4. 組織範圍控制的實現方式。統一管理可以簡化邏輯，但需要妥善設計資料表結構。組織階層關係可以使用自我關聯，但需要考慮查詢效能。

#### 7.1.3 功能運作原理

**核心邏輯的思考過程：**

權限管理系統的核心是統一管理使用者、角色、組織，並根據使用者的組織範圍控制資料存取。系統採用以下邏輯：

1. **統一資料模型：** 使用 permission_management 統一儲存使用者、角色、組織的資料，使用 type 欄位區分
2. **組織階層關係：** 使用 parent_id 建立組織階層關係，形成樹狀結構
3. **權限控制：** 使用 PermissionAttribute 檢查使用者權限，使用 OrganizationScope 過濾資料範圍
4. **檢視表輔助：** 使用 vw_permission_user、vw_permission_role、vw_permission_organization 提供特定類型的查詢介面

**運作流程：**

| 步驟 | 說明 | 設計考量 | 可能的替代方案 |
|------|------|----------|---------------|
| 1. 使用者登入 | 驗證使用者身份，載入使用者資訊 | 使用 Cookie 驗證，載入使用者資訊到 Claims | 使用 JWT，但 Cookie 更適合 Web 應用 |
| 2. 載入使用者權限 | 根據使用者的角色載入權限資訊 | 查詢使用者的角色，載入角色的導航權限 | 每次都查詢資料庫，但可以快取權限資訊 |
| 3. 檢查權限 | 使用 PermissionAttribute 檢查使用者是否有權限存取功能 | 在 Controller 層檢查權限，未授權時返回 AccessDenied | 在 Service 層檢查權限，但 Controller 層檢查更早，效能更好 |
| 4. 過濾資料範圍 | 使用 OrganizationScope 過濾資料，只顯示使用者組織範圍內的資料 | 在 Service 層過濾資料，確保資料安全性 | 在 Repository 層過濾資料，但 Service 層過濾更靈活 |

**設計模式與原則的應用：**

- **Repository Pattern：** 使用 Repository 層封裝資料存取邏輯，降低耦合度
- **Service Layer Pattern：** 使用 Service 層處理業務邏輯，保持 Controller 的簡潔
- **Attribute Pattern：** 使用 PermissionAttribute 統一檢查權限，降低程式碼重複
- **Scope Pattern：** 使用 OrganizationScope 統一過濾資料範圍，確保資料安全性

### 7.2 檔案管理系統

#### 7.2.1 功能設計背景

**需要解決的問題：**

企業需要一個檔案管理系統來管理檔案的上傳、下載、分享、預覽等。傳統的檔案伺服器方式缺乏權限控制與管理功能。系統需要支援多使用者檔案隔離、檔案分享功能、檔案預覽功能、資料夾管理等。

**問題的複雜度：**

檔案管理涉及多個面向：

1. **多使用者檔案隔離：** 每個使用者有自己的檔案儲存空間，檔案之間隔離，需要確保使用者無法存取其他使用者的檔案
2. **檔案分享功能：** 使用者可以將檔案分享給其他使用者，需要管理分享對象與權限
3. **檔案元資料管理：** 檔案元資料儲存在資料庫，實際檔案儲存在檔案系統，需要確保兩者的一致性
4. **權限控制：** 檔案的存取權限需要根據擁有者與分享對象判斷

#### 7.2.2 設計決策的思考過程

**設計目標：**

建立一個安全且靈活的檔案管理系統，支援多使用者隔離、檔案分享、權限控制等功能。系統應該提供良好的使用者體驗，同時確保資料安全性。

**設計方案比較：**

| 方案 | 優點 | 缺點 | 適用場景 | 為什麼不採用 |
|------|------|------|----------|-------------|
| 混合儲存（元資料在資料庫，檔案在檔案系統） | • 利用資料庫查詢能力<br>• 避免資料庫儲存大檔案<br>• 支援檔案分享與權限控制 | • 需要確保兩者一致性<br>• 備份較複雜 | 需要查詢與權限控制功能 | - |
| 全部儲存在資料庫 | • 一致性保證<br>• 備份簡單 | • 資料庫體積大<br>• 查詢效能差<br>• 不適合大檔案 | 檔案較小，不需要複雜查詢 | 本專案檔案可能較大，且需要複雜查詢與權限控制 |
| 全部儲存在檔案系統 | • 簡單直接<br>• 效能好 | • 無法查詢<br>• 無法權限控制<br>• 難以管理 | 檔案較少，不需要查詢與權限控制 | 本專案需要查詢與權限控制功能 |

**最終設計的考量：**

| 設計決策點 | 選擇的方案 | 為什麼這樣選擇 | 犧牲了什麼 | 如何權衡 |
|-----------|----------|--------------|-----------|----------|
| 儲存方式 | 元資料在資料庫，檔案在檔案系統 | 利用資料庫查詢能力，避免資料庫儲存大檔案 | 需要確保兩者一致性，備份較複雜 | 查詢與權限控制比一致性更重要，可以透過程式邏輯確保一致性 |
| 使用者隔離 | 每個使用者有自己的儲存目錄 | 確保不同使用者的檔案隔離，提升安全性 | 路徑管理較複雜 | 資料安全性比路徑管理複雜度更重要 |
| 虛擬路徑 | 使用者看到虛擬路徑，實際路徑隱藏 | 提供更友善的使用者體驗，隱藏實際檔案系統結構 | 需要路徑轉換邏輯 | 使用者體驗比路徑轉換邏輯更重要 |

**從這個設計中學到的模式：**

當設計檔案管理系統時，應該考慮：1. 儲存方式的選擇（資料庫 vs. 檔案系統 vs. 混合）2. 使用者隔離的實現方式 3. 檔案分享與權限控制的設計 4. 檔案元資料與實際檔案的一致性保證。混合儲存可以結合資料庫與檔案系統的優點，但需要確保兩者一致性。

### 7.3 AI 功能整合

#### 7.3.1 功能設計背景

系統需要整合多種 AI 功能：物件辨識（ONNX 模型）、圖像生成（DALL-E 3）、向量搜尋（Qdrant）等。這些功能需要與系統的其他功能（如檔案管理、權限控制）整合。

**問題的複雜度：**

1. **模型管理：** ONNX 模型檔案較大，需要管理模型的下載、初始化、健康檢查等

3. **向量資料庫整合：** Qdrant 向量資料庫需要連線設定、集合管理、向量搜尋等
4. **模組化設計：** AI 功能應該封裝為獨立模組，可以獨立測試與維護

#### 7.3.2 設計決策的思考過程

**設計目標：**

建立一個模組化且可擴展的 AI 功能整合架構，支援多種 AI 功能，同時確保系統的可維護性與可測試性。

**設計方案比較：**

| 方案 | 優點 | 缺點 | 適用場景 | 為什麼不採用 |

| 外部服務（微服務） | • 完全獨立<br>• 可獨立部署 | • 複雜度高<br>• 需要服務間通訊 | 大型分散式系統 | 專案規模尚未達到微服務的需求，模組化設計已足夠 |

**最終設計的考量：**

| 設計決策點 | 選擇的方案 | 為什麼這樣選擇 | 犧牲了什麼 | 如何權衡 |
|-----------|----------|--------------|-----------|----------|
| 模組組織 | 使用 IntelligentSystems 目錄組織 AI 模組 | 模組化設計，每個模組獨立，易於測試與維護 | 需要模組間整合 | 模組化設計可以提升可維護性與可擴展性 |
| 模型管理 | 使用 ModelHealthChecker 管理模型健康狀態 | 自動檢查模型是否存在，自動下載，確保 AI 功能可用 | 啟動時需要下載模型，可能影響啟動時間 | AI 功能的可用性比啟動時間更重要 |

| 配置管理 | 使用 appsettings.json 管理配置 | 集中管理配置，易於調整 | 敏感資訊可能暴露 | 可以使用環境變數或秘密管理工具管理敏感資訊 |

**從這個設計中學到的模式：**

## 8. 資料處理規則

### 8.1 權限資料驗證規則

#### 規則設計的問題識別

**需要解決的問題：**

權限資料的正確性與完整性是系統運作的基礎。如果權限資料不正確或不完整，會影響使用者的存取權限，導致安全問題或功能無法使用。系統需要確保所有權限資料都符合業務規則與資料格式要求。

**問題的複雜度：**

權限資料驗證涉及多個面向：

1. **必填欄位檢查：** 不同類型的權限實體（使用者、角色、組織）有不同的必填欄位要求
2. **唯一性驗證：** 使用者名稱、角色名稱、組織名稱等欄位需要確保唯一性
3. **組織階層關係驗證：** 組織的 parent_id 必須指向存在的組織，且不能形成循環引用
4. **角色權限格式驗證：** 角色的導航權限以字串陣列或 JSON 儲存，需要驗證格式正確性

#### 規則設計的思考過程

**設計目標：**

確保所有權限資料在儲存前都經過完整的驗證，包括必填欄位、唯一性、組織階層關係等。驗證失敗時應該返回明確的錯誤訊息，協助使用者修正輸入。

**最終設計的考量：**

| 驗證類型 | 判斷方式 | 設計原因 | 判斷方式的選擇 | 其他可能的判斷方式 |
|----------|----------|----------|---------------|------------------|
| 必填欄位 | 檢查欄位是否為空或 null | 缺失必填欄位會導致資料不完整 | 在 Repository 層檢查，統一處理 | 使用 Data Annotations，但無法根據類型動態檢查 |
| 唯一性驗證 | 查詢既有資料，檢查是否有重複 | 確保使用者名稱、角色名稱、組織名稱等唯一 | 在 Repository 層查詢，確保資料一致性 | 使用資料庫唯一約束，但應用程式層面驗證可以提供更清楚的錯誤訊息 |
| 組織階層關係 | 檢查 parent_id 是否指向存在的組織，檢查是否形成循環引用 | 組織階層關係錯誤會導致查詢失敗或邏輯錯誤 | 在 Repository 層遞迴檢查，確保階層關係正確 | 使用資料庫觸發器，但應用程式層面驗證可以提供更清楚的錯誤訊息 |

**從權限驗證設計中學到的模式：**

權限資料驗證應該：1. 在 Repository 層進行驗證，確保所有資料存取都經過驗證 2. 驗證應該考慮不同類型的權限實體（使用者、角色、組織）有不同的驗證規則 3. 驗證失敗時應該返回明確的錯誤訊息，協助使用者修正輸入 4. 組織階層關係驗證需要防止循環引用，確保階層關係的正確性。

### 8.2 檔案元資料與實際檔案的一致性規則

#### 規則設計的問題識別

**需要解決的問題：**

檔案管理的元資料儲存在資料庫（file_manager），實際檔案儲存在檔案系統。如果兩者不一致，會導致查詢到不存在的檔案或檔案系統中存在但資料庫中沒有記錄的檔案，影響系統的正確性與效能。

**問題的複雜度：**

檔案一致性涉及多個面向：

1. **新增檔案時的一致性：** 檔案上傳時需要同時在資料庫與檔案系統中建立記錄
2. **刪除檔案時的一致性：** 檔案刪除時需要同時刪除資料庫記錄與檔案系統中的檔案
3. **重新命名檔案時的一致性：** 檔案重新命名時需要同時更新資料庫記錄與檔案系統中的檔案名稱
4. **檔案移動時的一致性：** 檔案移動時需要同時更新資料庫記錄與檔案系統中的檔案位置

#### 更新邏輯的設計思考

**更新邏輯的設計目標：**

確保檔案元資料與實際檔案始終保持一致，避免不一致造成的問題。更新邏輯應該原子化，確保要麼全部成功，要麼全部失敗。

**更新邏輯的設計方案比較：**

| 方案 | 優點 | 缺點 | 適用場景 | 為什麼不採用 |
|------|------|------|----------|-------------|
| 先更新資料庫，後更新檔案系統 | • 資料庫更新快速<br>• 可以立即查詢 | • 檔案系統操作失敗會導致不一致<br>• 需要回滾機制 | 檔案系統操作較可靠 | 檔案系統操作可能失敗，需要回滾機制 |
| 先更新檔案系統，後更新資料庫 | • 檔案系統操作失敗不會影響資料庫 | • 資料庫更新失敗會導致檔案系統中存在但資料庫中沒有記錄的檔案 | 檔案系統操作較不可靠 | 資料庫更新失敗會導致孤兒檔案 |
| 使用交易機制 | • 可以確保原子性<br>• 失敗時自動回滾 | • 檔案系統操作無法納入資料庫交易<br>• 需要額外的補償機制 | 需要強一致性保證 | 檔案系統操作無法納入資料庫交易，無法實現真正的原子性 |

**最終設計的考量：**

系統採用先更新資料庫，後更新檔案系統的方式，如果檔案系統操作失敗，使用補償機制刪除資料庫記錄。這是因為：

1. **資料庫更新快速：** 資料庫更新比檔案系統操作快速，可以立即反映變更
2. **補償機制：** 檔案系統操作失敗時，使用補償機制刪除資料庫記錄，確保一致性
3. **錯誤處理：** 檔案系統操作失敗時返回錯誤訊息，使用者可以重試

**從檔案一致性設計中學到的模式：**

檔案一致性管理應該：1. 採用先更新資料庫，後更新檔案系統的方式，確保資料庫始終反映最新狀態 2. 檔案系統操作失敗時使用補償機制，確保一致性 3. 定期檢查檔案一致性，發現並修復不一致的問題 4. 提供檔案一致性檢查工具，協助管理者維護系統。

## 9. 故障排除與常見問題

### 9.1 錯誤日誌位置

| 錯誤類型 | 記錄位置 | 識別方式 | 說明 |
|----------|----------|----------|------|
| 系統例外錯誤 | Hnbdata.error_logs | `EventName` 欄位包含功能名稱或例外類型 | 系統運行過程中的例外錯誤，記錄詳細的堆疊追蹤資訊 |
| 存取記錄 | Hnbdata.access_records | `CreatedDate`、`CreatedUserName` 等欄位 | 記錄使用者的存取行為，用於安全審計 |
| 封鎖 IP 記錄 | Hnbdata.blocked_ips | `IpAddress`、`CreatedDate` 等欄位 | 記錄被封鎖的 IP 地址，用於 IP 安全控制 |
| AI 功能日誌 | HnbHnbBackoffice.ai_log、vw_ai_log | `CreatedDate`、`EventName` 等欄位 | 記錄 AI 功能的執行記錄，包含 API 呼叫、模型載入等 |
| 硬體監控資料 | HnbHnbBackoffice.hardware_monitoring、vw_hardware_monitoring | `CreatedDate`、`is_active` 等欄位 | 記錄硬體監控資料，包含 CPU、GPU、記憶體等使用率 |

**錯誤日誌關鍵字/識別對照：**

| 關鍵字/識別碼 | 模組/功能 | 說明 |
|--------------|----------|------|
| `Backoffice/Dashboard` | Dashboard 模組 | 儀表板相關錯誤 |
| `Backoffice/Authorize` | Authorize 模組 | 登入/授權相關錯誤 |
| `Backoffice/PermissionManagement` | PermissionManagement 模組 | 權限管理相關錯誤 |
| `Backoffice/FileManager` | FileManager 模組 | 檔案管理相關錯誤 |
| `ModelHealthChecker` | IntelligentSystems 模組 | 模型健康檢查相關錯誤 |

**本專案的錯誤日誌設計理念：**

本專案將錯誤日誌分散記錄到不同的資料表（error_logs、access_records、blocked_ips、ai_log 等），使用 `EventName` 或功能名稱標識錯誤來源。這樣的設計使得錯誤日誌分類清晰，方便查詢與分析。每個模組使用不同的錯誤標識，例如 Dashboard 使用 `Backoffice/Dashboard`。錯誤日誌記錄包含 `CreatedDate`、`CreatedUserName`、詳細錯誤訊息等欄位，提供完整的錯誤資訊。

### 9.2 錯誤日誌查詢方式

1. **系統例外錯誤（Hnbdata.error_logs）：**
   - 可依 `EventName`、`CreatedDate`、`CreatedUserName` 等條件篩選
   - 用於查詢特定功能或時間範圍的錯誤記錄
   - **專案中的查詢技巧：** 使用 `EventName` 欄位篩選特定模組的錯誤（例如 `Backoffice/FileManager`），使用 `CreatedDate` 篩選特定時間範圍的錯誤，使用 `CreatedUserName` 篩選特定使用者的錯誤。錯誤訊息中通常包含詳細的堆疊追蹤資訊，可以快速定位問題程式碼。

2. **AI 功能日誌（HnbHnbBackoffice.ai_log、vw_ai_log）：**
   - 使用 `EventName`、`CreatedDate` 等欄位篩選
   - 用於查詢 AI 功能的執行記錄，包含 API 呼叫、模型載入等
   - **專案中的查詢技巧：** 使用 `EventName` 篩選特定 AI 功能的記錄（例如物件辨識、圖像生成），使用 `CreatedDate` 篩選特定時間範圍的記錄。日誌中通常包含 API 回應、模型狀態等資訊，可以幫助診斷 AI 功能問題。

3. **硬體監控資料（HnbHnbBackoffice.hardware_monitoring、vw_hardware_monitoring）：**
   - 使用 `CreatedDate`、`is_active` 等欄位篩選
   - 用於查詢硬體監控資料，分析系統效能問題
   - **專案中的查詢技巧：** 使用 `CreatedDate` 篩選特定時間範圍的監控資料，使用 `is_active` 篩選啟用的監控項目。監控資料包含 CPU、GPU、記憶體等使用率，可以幫助診斷效能問題。

### 9.3 錯誤定位的思考方法

**錯誤排查的思維框架：**

錯誤排查應該遵循從外到內的思維框架：1. 先確認錯誤的徵兆與範圍（是特定功能、特定使用者、特定時間等）2. 再查看錯誤日誌，了解錯誤的詳細資訊 3. 最後追蹤到具體的程式碼位置，分析錯誤的根本原因。這樣的思維框架可以快速定位問題，避免盲目的排查。

**錯誤排查流程的設計：**

1. **前端/使用者介面錯誤：**
   - **為什麼先檢查這裡：** 使用者最直接感受到的是前端錯誤，例如頁面無法載入、按鈕無反應等。前端錯誤通常是後端問題的表現，但前端錯誤的徵兆最明顯，可以先從前端開始排查
   - **如何快速判斷：** 檢查瀏覽器的開發者工具（F12），查看 Console 是否有 JavaScript 錯誤、Network 是否有請求失敗、Response 是否有錯誤訊息
   - **常見的誤判情況：** 前端顯示的錯誤可能是後端錯誤的表現，例如後端回傳錯誤 JSON，前端顯示錯誤訊息。應該檢查 Network 中的 Response，確認錯誤的真正來源
   - **排查效率的考量：** 前端錯誤的徵兆明顯，可以快速判斷問題是否在前端，但需要注意前端錯誤可能是後端問題的表現

2. **後端/Controller 層錯誤：**
   - **為什麼需要檢查這裡：** Controller 層是請求處理的第一個點，負責輸入驗證、流程協調等。如果 Controller 層有問題，例如參數驗證失敗、路由錯誤等，會導致請求無法正確處理
   - **如何快速判斷：** 檢查 Controller 的日誌記錄，查看是否有例外錯誤。使用 ASP.NET Core 的開發環境，可以查看詳細錯誤頁面，了解例外錯誤的詳細資訊
   - **常見的誤判情況：** Controller 層的錯誤可能是 Service 層或 Repository 層錯誤的表現，例如 Service 層拋出例外，Controller 層記錄錯誤。應該追蹤到 Service 層，了解錯誤的根本原因
   - **排查效率的考量：** Controller 層的錯誤通常有清楚的錯誤訊息，可以快速定位問題，但需要追蹤到 Service 層或 Repository 層，了解錯誤的根本原因

3. **資料庫/資料層錯誤：**
   - **為什麼最後檢查這裡：** 資料庫錯誤通常是應用程式錯誤的最終表現，例如查詢失敗、資料不存在、連線失敗等。資料庫錯誤的排查需要檢查連線字串、資料表結構、資料內容等，相對複雜
   - **如何快速判斷：** 檢查資料庫連線是否正常，檢查 SQL 查詢是否正確，檢查資料表是否存在，檢查資料是否存在。使用 PostgreSQL 管理工具直接查詢資料庫，確認資料狀態
   - **常見的誤判情況：** 資料庫錯誤可能是應用程式邏輯錯誤的表現，例如查詢條件錯誤導致查詢失敗。應該檢查應用程式的查詢邏輯，確認查詢條件是否正確
   - **排查效率的考量：** 資料庫錯誤的排查需要檢查多個層面，相對複雜，但資料庫錯誤通常是問題的最終表現，排查到資料庫層面可以找到問題的根本原因

**系統化排查的思維模式：**

系統化排查應該遵循：1. 先確認錯誤的徵兆與範圍 2. 再查看錯誤日誌，了解錯誤的詳細資訊 3. 追蹤到具體的程式碼位置，分析錯誤的根本原因 4. 驗證修復方案，確認問題已解決。這個思維模式可以應用到任何錯誤排查場景，確保排查過程系統化且高效。

### 9.4 常見問題的思考與解決

#### Q1：使用者無法登入系統

**問題的識別：**

使用者嘗試登入系統時，系統回傳「使用者名稱或密碼錯誤」或「帳號已被停用」等錯誤訊息。問題的徵兆是登入失敗，使用者無法存取系統。

**問題的根本原因分析：**

登入功能需要驗證使用者身份，檢查使用者是否存在、密碼是否正確、帳號是否啟用等。可能的原因包括：

1. **使用者名稱或密碼錯誤：** 使用者輸入的使用者名稱或密碼不正確
2. **帳號已被停用：** 使用者的帳號被停用，無法登入
3. **使用者不存在：** 資料庫中找不到該使用者
4. **資料庫連線失敗：** 無法連線到資料庫，導致無法查詢使用者資料
5. **Cookie 設定錯誤：** Cookie 設定錯誤，導致登入狀態無法保存

**解決方案的思考：**

- **方案一：** 在錯誤訊息中明確說明錯誤原因（使用者名稱錯誤、密碼錯誤、帳號停用等）
  - **優缺點：** 優點是可以幫助使用者快速了解問題，缺點是可能暴露系統資訊（例如使用者是否存在）
  - **為什麼選擇：** 這個方案可以幫助使用者快速了解問題並解決，雖然可能暴露系統資訊，但可以透過模糊化錯誤訊息來降低風險

- **方案二：** 統一錯誤訊息「使用者名稱或密碼錯誤」，不區分具體原因
  - **優缺點：** 優點是不會暴露系統資訊，缺點是使用者無法快速了解問題
  - **為什麼不選擇：** 這個方案雖然安全性較高，但會降低使用者體驗

**最終解決方案：**

在錯誤訊息中明確說明錯誤原因，但對於安全性敏感的資訊（例如使用者是否存在）使用模糊化的錯誤訊息。例如：使用者名稱錯誤時顯示「使用者名稱或密碼錯誤」，帳號停用時顯示「帳號已被停用，請聯絡管理員」。

**問題的預防思考：**

1. **提供密碼重設功能：** 如果使用者忘記密碼，可以透過密碼重設功能重新設定密碼
2. **帳號啟用提醒：** 帳號被停用時，系統可以發送通知提醒使用者或管理員
3. **登入嘗試限制：** 限制登入嘗試次數，防止暴力破解

**從這個問題中學到的經驗：**

1. **錯誤訊息應該明確但不暴露敏感資訊：** 錯誤訊息應該幫助使用者快速了解問題，但不應該暴露系統的敏感資訊（例如使用者是否存在）
2. **提供自助解決方案：** 對於常見問題（例如忘記密碼），應該提供自助解決方案，減少管理員的負擔
3. **安全性與使用者體驗的權衡：** 在設計錯誤訊息時，需要在安全性與使用者體驗之間權衡，選擇最適合的方案

#### Q2：AI 功能無法運作，提示「模型檔案不存在」

**問題的識別：**

使用者嘗試使用 AI 功能（例如物件辨識）時，系統回傳「模型檔案不存在」或「模型初始化失敗」等錯誤訊息。問題的徵兆是 AI 功能無法使用。

**問題的根本原因分析：**

AI 功能需要載入 ONNX 模型檔案才能運作。ModelHealthChecker 會在啟動時檢查模型是否存在，如果不存在會自動下載。可能的原因包括：

1. **模型檔案路徑錯誤：** `appsettings.json` 中的模型路徑設定錯誤，導致系統找不到模型檔案
2. **模型檔案下載失敗：** 模型檔案下載時網路連線失敗，導致下載失敗
3. **模型檔案損壞：** 模型檔案下載完成後損壞，導致無法載入
4. **模型檔案權限不足：** 模型檔案所在目錄的權限不足，導致無法讀取模型檔案
5. **ModelHealthChecker 未啟動：** ModelHealthChecker 未正常啟動，導致模型檢查與下載機制未運作

**解決方案的思考：**

- **方案一：** 在 ModelHealthChecker 中加強錯誤處理，提供詳細的錯誤訊息與修復建議
  - **優缺點：** 優點是可以幫助使用者快速了解問題並解決，缺點是增加了系統複雜度
  - **為什麼選擇：** 這個方案可以幫助使用者快速了解問題並解決，提升使用者體驗

- **方案二：** 提供模型檔案手動上傳功能，允許使用者手動上傳模型檔案
  - **優缺點：** 優點是提供了備援方案，缺點是需要額外的檔案上傳功能
  - **為什麼選擇：** 這個方案提供了備援方案，當自動下載失敗時，使用者可以手動上傳模型檔案

**最終解決方案：**

1. 在 ModelHealthChecker 中加強錯誤處理，提供詳細的錯誤訊息與修復建議
2. 提供模型檔案手動上傳功能，允許使用者手動上傳模型檔案
3. 在錯誤日誌中記錄模型檢查與下載的詳細資訊，協助診斷問題

**問題的預防思考：**

1. **模型檔案備份：** 將模型檔案備份到多個位置，確保模型檔案的可用性
2. **模型檔案完整性檢查：** 下載模型檔案後檢查檔案完整性，確保檔案未損壞
3. **模型檔案路徑驗證：** 在啟動時驗證模型檔案路徑，確保路徑正確

**從這個問題中學到的經驗：**

1. **自動化機制需要有備援方案：** 自動化的機制（例如模型自動下載）可能失敗，需要有備援方案（例如手動上傳）
2. **錯誤處理應該提供詳細資訊：** 錯誤處理應該提供詳細的錯誤訊息與修復建議，幫助使用者快速解決問題
3. **日誌記錄的重要性：** 詳細的日誌記錄可以幫助診斷問題，應該在關鍵步驟記錄詳細資訊

### 10.1 建置與部署流程

#### 10.1.1 建置

```bash
dotnet build HNB/Horizon_Nova/HNB/HNB.csproj
```

**本專案的建置考量：**

本專案使用 ASP.NET Core 8.0，需要在 .NET SDK 8.0 環境中建置。建置前需要確認：

1. **資料庫連線字串：** 確認 `appsettings.json` 中的資料庫連線字串正確
2. **AI 服務設定：** 確認 `appsettings.json` 中的 AI 服務設定（DALL-E 3 API Key、Qdrant 設定等）正確
3. **模型檔案路徑：** 確認 `appsettings.json` 中的模型檔案路徑設定正確
4. **儲存目錄權限：** 確認儲存目錄（`Areas/Backoffice/storage`）的權限設定正確，應用程式有讀寫權限

#### 10.1.2 發佈

```bash
dotnet publish HNB/Horizon_Nova/HNB/HNB.csproj -c Release -o ./publish
```

**本專案的發佈考量：**

發佈時需要注意：

1. **設定檔管理：** 發佈時需要區分開發環境與正式環境的設定檔（`appsettings.Development.json` vs. `appsettings.json`）

3. **模型檔案處理：** 模型檔案較大，不適合包含在發佈包中，應該在部署後由 ModelHealthChecker 自動下載

### 10.2 維護作業

#### 10.2.1 資料庫備份

**本專案中維護作業的目的：**

定期備份資料庫可以防止資料遺失，確保系統在發生問題時可以恢復。本專案使用兩個資料庫，需要分別備份。

**本專案中維護作業的背景：**

本專案使用 PostgreSQL 資料庫，可以使用 `pg_dump` 工具備份資料庫。備份應該包含兩個資料庫：Hnbdata 與 HnbHnbBackoffice。

| 步驟 | 操作 | 注意事項 | 專案特定的原因 |
|------|------|----------|---------------|
| 1 | 備份 Hnbdata 資料庫 | 使用 `pg_dump` 工具備份，儲存到安全位置 | 系統需要兩個資料庫，需要分別備份 |
| 2 | 備份 HnbHnbBackoffice 資料庫 | 使用 `pg_dump` 工具備份，儲存到安全位置 | 系統需要兩個資料庫，需要分別備份 |
| 3 | 備份檔案系統 | 備份 `Areas/Backoffice/storage` 目錄 | 檔案管理功能使用檔案系統儲存，需要備份檔案系統 |

**維護作業在專案中的影響：**

資料庫備份是維護作業，不會影響系統運作。但備份時可能會產生較大的檔案，需要確保儲存空間足夠。檔案系統備份時可能會影響檔案存取效能，建議在低峰時段執行。

#### 10.2.2 日誌清理

**本專案中維護作業的目的：**

定期清理舊日誌可以釋放資料庫空間，提升查詢效能。本專案的日誌分散在多個資料表，需要分別清理。

**本專案中維護作業的背景：**

本專案的日誌包括：error_logs、access_records、ai_log 等。這些日誌會隨著時間累積，需要定期清理。

| 步驟 | 操作 | 注意事項 | 專案特定的原因 |
|------|------|----------|---------------|
| 1 | 清理 error_logs | 刪除30天前的錯誤日誌，保留最近的日誌用於問題診斷 | 錯誤日誌用於問題診斷，保留最近的日誌即可 |
| 2 | 清理 access_records | 刪除30天前的存取記錄，保留最近的記錄用於安全審計 | 存取記錄用於安全審計，保留最近的記錄即可 |
| 3 | 清理 ai_log | 刪除30天前的 AI 功能日誌，保留最近的日誌用於問題診斷 | AI 功能日誌用於問題診斷，保留最近的日誌即可 |

**維護作業在專案中的影響：**

日誌清理會刪除資料，需要謹慎執行。建議先備份資料庫，確認備份成功後再執行清理。清理後需要檢查系統是否正常運作。

## 1. 專案概述

### 1.1 系統定位

Horizon Nova 是一個功能完整的企業級後台管理系統，提供全方位的管理工具與權限控制功能，包含使用者管理、角色管理、組織管理、檔案管理、系統監控、AI 功能等。

**系統需求背景：**

企業需要一個統一的後台管理系統來管理組織架構、使用者權限、系統資源等。傳統的多系統分散管理方式效率低、維護成本高、權限管理複雜。系統需要支援：

1. **完整的權限管理體系：** 支援使用者、角色、組織的多層級權限管理，並提供組織圖視覺化
2. **檔案管理功能：** 提供檔案上傳、下載、分享、預覽等功能，支援多使用者隔離
3. **系統監控與維護：** 提供硬體監控（CPU、GPU、記憶體、網路、儲存、電源）、日誌管理、快取清理等功能
4. **AI 功能整合：** 整合物件辨識（ONNX 模型）、圖像生成（DALL-E 3）、向量資料庫（Qdrant）等 AI 功能
5. **彈性的導航管理：** 支援動態側邊欄導航，可根據使用者權限顯示不同的導航項目

此外，系統需要提供良好的擴展性與維護性，支援多專案管理、團隊協作等功能。

**系統定位：**

本系統在整體架構中屬於企業級後台管理系統，作為其他系統的管理入口與基礎平台。系統的核心定位是：

1. **權限管理平台：** 統一管理所有系統的使用者、角色、組織與權限設定
2. **系統管理中心：** 提供系統監控、日誌管理、檔案管理等基礎功能
3. **AI 功能平台：** 整合各種 AI 功能（物件辨識、圖像生成、向量搜尋），提供統一的 AI 服務
4. **擴展平台：** 支援多專案管理、團隊協作等功能，為未來擴展提供基礎

系統採用 Areas 架構，主要分為 Backoffice（後台管理）與 NovaAPP（應用程式）兩個區域，目前以 Backoffice 為主。

#### ASP.NET Core 8.0 的選型分析

**問題識別：**

需要建立一個現代化的企業級後台管理系統，支援完整的權限管理、檔案管理、系統監控、AI 功能整合等。系統需要良好的擴展性、效能與維護性，並能夠整合各種外部服務（AI API、向量資料庫等）。

**方案評估：**

| 方案 | 優點 | 缺點 | 適用場景 | 不選擇的原因 |
|------|------|------|----------|-------------|
| ASP.NET Core 8.0 | • 現代化架構<br>• 跨平台支援<br>• 效能優異<br>• 長期支援<br>• 豐富的擴展生態 | • 需要學習新框架<br>• 設定較複雜 | 新專案、需要現代化架構 | - |
| ASP.NET MVC Framework | • 穩定成熟<br>• 團隊熟悉<br>• 設定簡單 | • 較舊的技術<br>• 只能在 Windows 運行<br>• 效能較差<br>• 長期支援有限 | 既有系統維護、Windows 限定環境 | 專案需要跨平台支援，且需要現代化的功能與效能 |
| Node.js / Express | • 非同步效能佳<br>• JavaScript 全端<br>• 生態豐富 | • 型別安全性較差<br>• 團隊熟悉度 | 前後端統一技術棧 | 團隊對 .NET 熟悉，且需要強型別支援 |

**決策依據：**

本專案是一個新建立的企業級後台管理系統，需要現代化的架構與良好的擴展性。ASP.NET Core 8.0 提供了：

1. **現代化架構：** 依賴注入、中介軟體、過濾器等現代化特性
2. **跨平台支援：** 可以在 Windows、Linux、macOS 運行，提升部署彈性
3. **效能優異：** 相比 ASP.NET MVC Framework 有顯著的效能提升
4. **豐富的擴展：** 可以輕鬆整合 AI 服務、向量資料庫等外部服務
5. **長期支援：** .NET 8 是長期支援版本，提供長期維護保證

**實際應用與調整：**

實際應用中，使用 Areas 架構劃分不同的功能區域（Backoffice、NovaAPP），每個 Area 有獨立的 Controllers、Views、Services、Repositories。使用依賴注入統一管理服務，使用中介軟體處理通用邏輯（錯誤處理、IP 安全檢查、硬體監控等）。這樣的設計提供了清晰的架構邊界，易於維護與擴展。

**可複用的思考模式：**

當選擇後端框架時，應該考慮：1. 專案的長期維護需求 2. 團隊的技術熟悉度 3. 系統的效能要求 4. 擴展性與整合需求。選擇現代化框架雖然有學習成本，但可以提升長期競爭力與維護性。

#### Entity Framework Core 8.0 與 PostgreSQL 的選型分析

**問題識別：**

系統需要存取兩個不同的資料庫：Hnbdata（專案資料、團隊成員、存取記錄等）與 HnbHnbBackoffice（後台管理相關資料，如權限、導航、檔案管理等）。每個資料庫有獨立的業務領域，資料模型獨立。系統需要選擇資料庫類型與 ORM 框架。

**方案評估：**

| 方案 | 優點 | 缺點 | 適用場景 | 不選擇的原因 |
|------|------|------|----------|-------------|
| EF Core 8.0 + PostgreSQL | • 強型別支援<br>• LINQ 查詢<br>• 遷移管理<br>• 跨平台<br>• 開源免費<br>• 效能優異 | • 學習成本較高<br>• 設定較複雜 | 現代化應用、需要 ORM 功能 | - |
| EF Core + SQL Server | • 強型別支援<br>• LINQ 查詢<br>• 與 Windows 整合良好<br>• 團隊熟悉 | • 授權成本高<br>• 跨平台支援較差 | Windows 限定環境、已有 SQL Server | 專案需要跨平台支援，且希望降低授權成本 |
| Dapper + PostgreSQL | • 效能較佳<br>• 靈活性高<br>• 學習曲線低 | • 需要手寫 SQL<br>• 型別安全性較低<br>• 無遷移功能 | 效能要求極高的場景 | 本專案需要型別安全與 LINQ 查詢，效能要求不極端 |
| Npgsql（直接使用） | • 效能最佳<br>• 完全控制 | • 需要大量手寫程式碼<br>• 維護成本高<br>• 容易出錯 | 極端效能要求的場景 | 開發效率與維護成本不划算 |

**決策依據：**

本專案選擇 PostgreSQL 作為資料庫，EF Core 8.0 作為 ORM 框架，主要考慮：

1. **跨平台支援：** PostgreSQL 可以在多種作業系統運行，與 ASP.NET Core 的跨平台特性匹配
2. **開源免費：** PostgreSQL 是開源資料庫，不需要授權成本，適合企業內部使用
3. **效能優異：** PostgreSQL 在大量資料處理與複雜查詢方面表現優異
4. **功能豐富：** PostgreSQL 支援 JSON、陣列、全文搜尋等進階功能，適合複雜的資料需求
5. **EF Core 整合：** EF Core 8.0 對 PostgreSQL 有良好的支援，提供強型別與 LINQ 查詢

使用 EF Core 而非 Dapper 的原因是：本專案需要強型別支援、LINQ 查詢、遷移管理等功能，可以大幅提升開發效率與程式碼品質。雖然效能略遜於 Dapper，但在可接受範圍內。

**實際應用與調整：**

實際應用中，每個資料庫使用獨立的 DbContext（HnbdataDbContext、HnbHnbBackofficeDbContext），在 `Program.cs` 中分別註冊。連線字串在 `appsettings.json` 中設定，使用 Npgsql 連線 PostgreSQL。這樣的設計確保了資料隔離，每個資料庫的資料模型互不干擾。

**可複用的思考模式：**

當選擇資料庫與 ORM 時，應該考慮：1. 跨平台需求 2. 授權成本 3. 效能要求 4. 團隊熟悉度 5. 功能需求。選擇開源資料庫可以降低授權成本，選擇現代化的 ORM 可以提升開發效率。

#### Areas 架構的選型分析

**問題識別：**

系統需要支援多個功能區域：Backoffice（後台管理）、NovaAPP（應用程式，未來可能擴展）。每個區域有獨立的業務邏輯、頁面與權限控制。需要一個清晰的架構來組織這些區域。

**方案評估：**

| 方案 | 優點 | 缺點 | 適用場景 | 不選擇的原因 |
|------|------|------|----------|-------------|
| Areas 架構 | • 清晰的功能邊界<br>• 易於組織大型專案<br>• 支援獨立的路由與權限控制<br>• ASP.NET Core 原生支援 | • 設定較複雜<br>• 需要理解 Areas 概念 | 大型專案、需要功能分區 | - |
| 單一 Controller/View 結構 | • 設定簡單<br>• 結構清晰 | • 不適合大型專案<br>• 難以管理多個功能區域 | 小型專案、單一功能區域 | 專案規模較大，且有明確的功能分區需求 |
| 微服務架構 | • 完全獨立<br>• 可獨立部署<br>• 技術棧可不同 | • 複雜度高<br>• 維護成本高<br>• 需要服務間通訊 | 大型分散式系統 | 專案規模尚未達到微服務的需求，Areas 架構已足夠 |

**決策依據：**

本專案採用 Areas 架構，主要考慮：

1. **功能邊界清晰：** Backoffice 與 NovaAPP 有不同的業務邏輯與使用者群體，需要清晰的邊界
2. **易於擴展：** 未來可以輕鬆新增新的 Area（如 Mobile、API 等），不影響既有功能
3. **獨立的路由與權限：** 每個 Area 可以有獨立的路由規則與權限控制
4. **符合 ASP.NET Core 最佳實踐：** Areas 是 ASP.NET Core 推薦的大型專案組織方式

雖然設定較複雜，但提供了清晰的架構邊界，長期來看可以提升可維護性。

**實際應用與調整：**

實際應用中，每個 Area 有獨立的 Controllers、Views、Services、Repositories。使用 BaseController 提供共用的功能（如側邊欄載入），使用 PermissionAttribute 提供統一的權限控制。路由設定在 `Program.cs` 中使用 `MapControllerRoute` 設定，支援 Areas 路由。

**可複用的思考模式：**

當組織大型專案時，應該考慮：1. 功能邊界的清晰度 2. 未來的擴展需求 3. 維護的便利性。Areas 架構可以提供清晰的功能邊界，但不應該過度使用，只有在有明確功能分區需求時才使用。

#### AI 功能整合的選型分析

**問題識別：**

系統需要整合多種 AI 功能：物件辨識（ONNX 模型）、圖像生成（DALL-E 3）、向量搜尋（Qdrant）。這些功能需要與系統的其他功能（如檔案管理、權限控制）整合。

**方案評估：**

| 方案 | 優點 | 缺點 | 適用場景 | 不選擇的原因 |
|------|------|------|----------|-------------|
| ONNX Runtime + DALL-E 3 API + Qdrant | • ONNX 模型可在本地運行<br>• DALL-E 3 API 提供高品質圖像生成<br>• Qdrant 提供高效能向量搜尋<br>• 各自獨立，可分別擴展 | • 需要整合多個系統<br>• 設定複雜 | 需要多種 AI 功能的場景 | - |
| 單一 AI 平台（如 Azure Cognitive Services） | • 統一管理<br>• 設定簡單<br>• 支援多種 AI 功能 | • 授權成本高<br>• 依賴特定平台<br>• 功能可能受限 | 預算充足、依賴特定平台 | 專案需要更靈活的整合方式，且希望降低授權成本 |
| 自行開發 AI 功能 | • 完全控制<br>• 無授權成本 | • 開發成本高<br>• 技術難度高<br>• 維護成本高 | 有 AI 專業團隊 | 專案不需要從零開發，使用現有工具更有效率 |

**決策依據：**

本專案選擇整合多個 AI 系統，主要考慮：

1. **功能需求多樣化：** 需要物件辨識、圖像生成、向量搜尋等多種功能，單一平台可能無法滿足所有需求
2. **成本考量：** ONNX 模型可以在本地運行，降低 API 呼叫成本。DALL-E 3 API 按使用量計費，成本可控
3. **技術靈活性：** 使用開源工具（ONNX Runtime、Qdrant）可以避免廠商鎖定，未來可以替換其他工具
4. **效能考量：** ONNX 模型在本地運行可以提供低延遲的辨識服務，適合即時應用

雖然需要整合多個系統，但提供了更好的功能選擇與成本控制。

**實際應用與調整：**

實際應用中，將 AI 功能封裝為獨立模組（IntelligentSystems），每個模組有自己的配置、API 與服務。使用 ModelHealthChecker 管理 ONNX 模型的健康狀態，自動下載與初始化模型。使用 Dependency Injection 注入 AI 模組到需要使用的服務中。

**可複用的思考模式：**

當整合多個外部服務時，應該考慮：1. 功能需求的完整性 2. 成本效益 3. 技術靈活性 4. 維護成本。選擇開源工具可以降低授權成本，但需要考慮維護成本。封裝為獨立模組可以提升可維護性與可測試性。

**整體架構設計的演進：**

**初始設計思路：**

最初設計時，考慮將所有功能放在單一的 Controller/View 結構中，這樣的設計雖然簡單，但隨著功能增加，會導致結構混亂，難以維護。

**遇到的問題與調整：**

- **問題一：** 功能模組過多，Controller 與 View 結構混亂
  - **問題的根本原因：** 沒有明確的功能邊界，所有功能混在一起
  - **第一版的不足：** 可維護性差，難以擴展，功能間耦合度高

- **調整方案：**
  - **調整思路：** 採用 Areas 架構，將功能劃分為不同的區域（Backoffice、NovaAPP）
  - **方案比較：** 考慮過使用多專案架構，但 Areas 架構提供了更好的整合性，且可以共用基礎設施（認證、授權等）
  - **改進後的優勢：** 每個 Area 有清晰的邊界，易於維護與擴展。使用 BaseController 提供共用功能，降低重複程式碼

**當前架構的設計原則：**

1. **功能分區原則：** 使用 Areas 架構劃分不同的功能區域，每個 Area 有獨立的業務邏輯
2. **依賴注入原則：** 所有服務透過 Dependency Injection 注入，降低耦合度
3. **分層架構原則：** Controller → Service → Repository 三層架構，明確職責分離
4. **模組化原則：** AI 功能封裝為獨立模組（IntelligentSystems），可以獨立測試與維護

**未來可能的改進方向：**

1. 考慮將 AI 模組獨立為微服務，提供更好的擴展性與效能
2. 引入 CQRS 模式，分離讀寫操作，提升查詢效能
3. 考慮使用 GraphQL 提供更靈活的 API 查詢介面

**本專案的結構設計理念：**

本專案採用 Areas 架構組織大型功能區域，使用分層架構（Controller → Service → Repository）確保職責分離。AI 功能封裝為獨立的 IntelligentSystems 模組，可以獨立測試與維護。使用 Extensions 組織依賴注入的設定，使用 Filters 與 Middleware 處理通用邏輯。

**專案特定的組織原則：**

1. **Areas 優先原則：** 主要功能都放在 Areas 下，根目錄的 Controllers 只保留必要的控制器（如 ErrorController）
2. **分層架構原則：** Controller → Service → Repository 三層架構，每層都有明確的職責
3. **模組化原則：** AI 功能、工具類別、中介軟體等都以模組方式組織，提升可重用性
4. **Partial View 組織：** Views 使用 Partials 組織，每個頁面有自己的 Partials 目錄，Modal 放在 Modal 子目錄中

**重要檔案的特殊用途：**

1. **Program.cs：** 應用程式的啟動與設定入口，註冊所有服務、中介軟體、路由等
2. **ServiceExtensions.cs：** 統一管理所有依賴注入設定，包含 Services、Repositories、Utilities、AI 模組等
3. **BaseController：** 提供所有後台管理控制器的共用功能（如側邊欄載入），並套用權限控制
4. **ModelHealthChecker：** 管理 ONNX 模型的健康狀態，自動下載與初始化模型，確保 AI 功能可用

1. **Hnbdata：** 儲存專案相關的資料（project、project_tags、team_member）與系統日誌（error_logs、access_records、blocked_ips）
2. **HnbHnbBackoffice：** 儲存後台管理相關的資料（權限、導航、檔案管理、硬體監控、AI 配置等）

每個資料庫使用獨立的 DbContext，確保資料隔離與連線管理。檔案管理使用混合儲存方式：檔案元資料儲存在資料庫（file_manager），實際檔案儲存在檔案系統（Areas/Backoffice/storage）。

**專案資料來源間的關係：**

1. **Hnbdata 與 HnbHnbBackoffice：** 兩個資料庫之間沒有直接的資料庫層級關聯，所有關聯都在應用程式層面處理。例如：檔案管理記錄（file_manager）可能與專案（project）邏輯關聯，但關聯關係透過應用程式邏輯處理。

2. **檔案系統與資料庫：** 檔案管理的元資料（檔案名稱、路徑、大小、擁有者等）儲存在資料庫（file_manager），實際檔案儲存在檔案系統。這樣的設計可以：
   - 利用資料庫的查詢能力（如搜尋、排序、權限控制）
   - 利用檔案系統的儲存效能（大檔案不佔用資料庫空間）
   - 支援檔案分享與權限控制（透過資料庫記錄擁有者與分享對象）

3. **AI 服務與資料庫：** AI 配置（ai_config）與 AI 日誌（ai_log）儲存在資料庫，但 AI 模型檔案（ONNX）儲存在檔案系統。AI 服務透過 HTTP 或 SDK 連線到外部服務（DALL-E 3 API、Qdrant），不直接操作資料庫。

邏輯關聯在應用程式層面處理，透過欄位值匹配（如 ID）建立關聯。側邊欄導航與權限的關聯使用字串陣列或 JSON 儲存多個權限 ID，這樣的設計提供了更靈活的權限控制，但需要在應用程式層面處理關聯邏輯。

**專案中關聯的影響：**

1. **組織階層關係：** permission_management 使用 parent_id 建立組織階層關係，可以建立多層級的組織架構。查詢組織時需要遞迴查詢，或使用檢視表簡化查詢。

2. **檔案擁有者關係：** file_manager 透過 owner_id 關聯到 permission_management（使用者），建立檔案擁有者關係。檔案分享功能需要額外的資料表或欄位來記錄分享對象。

3. **跨資料庫關聯：** team_member 與 permission_management 分屬不同資料庫，關聯關係在應用程式層面處理。查詢團隊成員時需要同時查詢兩個資料庫，在應用程式層面組合資料。

### 5.3 關鍵資料表設計說明

#### HnbHnbBackoffice.permission_management

**本專案中此資料表的設計目的：**

permission_management 資料表是權限管理的核心資料表，統一儲存使用者、角色、組織的資料。使用 `type` 欄位區分不同的實體類型（使用者、角色、組織），這樣的設計可以統一管理權限，簡化權限控制的邏輯。

**專案特定的設計考量：**

- **type 欄位：** 使用字串區分實體類型（如 "user"、"role"、"organization"），統一管理不同類型的權限實體
- **parent_id 欄位：** 用於建立組織階層關係，組織可以有上層組織，形成樹狀結構
- **統一權限模型：** 使用者、角色、組織都使用相同的資料表，簡化權限管理邏輯
- **檢視表輔助：** 使用 `vw_permission_user`、`vw_permission_role`、`vw_permission_organization` 等檢視表提供特定類型的查詢介面

**本專案中資料的生命週期：**

1. **新增：** 透過權限管理頁面新增使用者、角色或組織，系統驗證必填欄位與唯一性
2. **修改：** 透過權限管理頁面修改資料，系統驗證後更新
3. **刪除：** 使用邏輯刪除或實體刪除（依業務規則而定），刪除前需要檢查是否有其他資料關聯
4. **查詢：** 透過檢視表查詢特定類型的資料，支援權限範圍過濾（OrganizationScope）

**專案中的使用情境：**

- **使用者管理：** 查詢、新增、修改、刪除使用者，設定使用者的角色與組織歸屬
- **角色管理：** 查詢、新增、修改、刪除角色，設定角色的導航權限
- **組織管理：** 查詢、新增、修改、刪除組織，建立組織階層關係，查詢組織圖

#### HnbHnbBackoffice.file_manager

**本專案中此資料表的設計目的：**

file_manager 資料表儲存檔案管理的元資料，包括檔案名稱、路徑、大小、擁有者、分享對象等。實際檔案儲存在檔案系統，資料庫只儲存元資料，這樣的設計可以：

1. **利用資料庫的查詢能力：** 透過 SQL 查詢檔案、排序、篩選
2. **支援檔案分享與權限控制：** 透過資料庫記錄擁有者與分享對象，實現檔案分享功能
3. **避免資料庫儲存大檔案：** 實際檔案儲存在檔案系統，不佔用資料庫空間

**專案特定的設計考量：**

- **virtual_path 欄位：** 虛擬路徑，用於使用者介面顯示，與實際檔案路徑可能不同
- **actual_path 欄位：** 實際檔案路徑，對應檔案系統中的路徑
- **owner_id 欄位：** 檔案擁有者（使用者 ID），用於權限控制
- **shared_with 欄位：** 分享對象（可能為 JSON 或關聯資料表），記錄可以存取此檔案的使用者

**本專案中資料的生命週期：**

1. **新增：** 使用者上傳檔案時，系統儲存實際檔案到檔案系統，並在資料庫中建立元資料記錄
2. **修改：** 修改檔案名稱、路徑、分享對象等元資料
3. **刪除：** 執行邏輯刪除或實體刪除（依業務規則而定），實際檔案可能立即刪除或延後刪除
4. **查詢：** 根據使用者權限查詢檔案，支援路徑篩選、檔案類型篩選等

**專案中的使用情境：**

- **檔案上傳：** 使用者上傳檔案，系統儲存到檔案系統並建立元資料
- **檔案查詢：** 根據路徑查詢檔案清單，顯示檔案資訊（名稱、大小、修改時間等）
- **檔案分享：** 設定檔案的分享對象，允許其他使用者存取
- **檔案下載：** 根據權限檢查後提供檔案下載

#### HnbHnbBackoffice.hardware_monitoring

**本專案中此資料表的設計目的：**

hardware_monitoring 資料表儲存硬體監控資料，包括 CPU、GPU、記憶體、網路、儲存、電源等監控資訊。這些資料由 HardwareMonitoringMiddleware 定期收集並寫入資料庫，用於系統健康狀態監控。

**專案特定的設計考量：**

- **is_active 欄位：** 監控服務是否啟用，用於控制監控功能的開關
- **各種監控欄位：** cpu_usage_percent、memory_usage_percent、gpu_usage_percent 等，儲存各種硬體資源的使用率
- **監控時間：** 記錄監控資料的收集時間，用於分析趨勢

**本專案中資料的生命週期：**

1. **自動收集：** HardwareMonitoringMiddleware 定期收集硬體監控資料並寫入資料庫
2. **查詢顯示：** 透過 Dashboard 或 Settings 頁面查詢並顯示最新的監控資料
3. **歷史資料：** 保留歷史監控資料，用於分析趨勢與問題診斷

**專案中的使用情境：**

- **系統狀態監控：** Dashboard 顯示系統的整體健康狀態（CPU、記憶體使用率等）
- **硬體監控管理：** Settings 頁面提供硬體監控的詳細資訊與設定
- **問題診斷：** 透過歷史監控資料分析系統效能問題

### 5.4 資料庫直接操作說明

1. **連線資訊：** 參考 3.2 節連線設定
2. **資料表關聯關係：** 參考 5.2 節
3. **注意事項：**
   - 手動修正資料前建議先備份
   - 修改 permission_management 資料時，需要確認組織階層關係的正確性
   - 刪除檔案記錄時，需要確認實際檔案是否也需要刪除
   - 修改權限設定時，需要確認對應的使用者、角色、組織關係

**本專案中需要注意這些事項的原因：**

1. **組織階層關係的重要性：** permission_management 使用 parent_id 建立組織階層關係，如果階層關係錯誤，會導致組織圖顯示錯誤、權限範圍計算錯誤等問題。修改組織資料時需要特別注意階層關係。

2. **檔案元資料與實際檔案的一致性：** file_manager 只儲存檔案元資料，實際檔案儲存在檔案系統。如果只刪除資料庫記錄而不刪除實際檔案，會造成檔案系統空間浪費。如果只刪除實際檔案而不刪除資料庫記錄，會導致查詢到不存在的檔案。

3. **權限關係的完整性：** permission_management 統一管理使用者、角色、組織，修改任何一個實體時都需要確認對應的關係（如使用者的角色歸屬、角色的導航權限等）是否正確。

### 6.2 各模組詳細說明

#### Dashboard（儀表板）模組

**本專案的需求背景：**

使用者登入後需要一個總覽頁面，快速了解系統狀態、存取常用功能、查看近期活動等。傳統的分散式頁面設計讓使用者難以快速掌握系統整體狀況。

**主要功能：**

- **功能一：** 系統狀態顯示 - 顯示系統的整體健康狀態（伺服器狀態、CPU 使用率、記憶體使用率等）
- **功能二：** 快速操作 - 提供常用功能的快速存取連結
- **功能三：** 近期活動顯示 - 顯示最近的系統活動或專案資訊
- **功能四：** 歡迎訊息 - 顯示使用者的歡迎訊息與基本資訊

**查詢資料來源：**

- HnbHnbBackoffice.hardware_monitoring：硬體監控資料（CPU、記憶體使用率等）
- Hnbdata.project：專案資料（用於顯示近期活動）

**專案中這些資料的用途：**

- hardware_monitoring：提供系統狀態的即時資訊，幫助使用者快速了解系統健康狀態
- project：顯示專案列表或最近的專案活動，幫助使用者快速了解專案狀況

**異動資料來源：**

無（此模組只提供查詢功能，不負責新增或修改資料）

**本專案的操作邏輯：**

1. **頁面載入：** 使用者登入後進入 Dashboard，系統載入最新的硬體監控資料與專案資料，顯示在頁面上
2. **即時更新：** 系統狀態可能需要定期更新（透過 AJAX），但本模組目前可能只顯示載入時的資料

**專案特定的規則：**

- **資料載入時機：** Dashboard 頁面載入時才查詢硬體監控資料，不提供即時自動更新。如果需要即時更新，需要在前端使用定時器呼叫 AJAX 更新。

**規則原因：**

Dashboard 是總覽頁面，主要目的是快速了解系統狀態，不需要即時更新。如果使用者需要即時的監控資料，可以前往 Settings 頁面查看詳細的硬體監控資訊。

**本專案中模組間的關聯：**

Dashboard 模組依賴 Settings 模組提供硬體監控資料，依賴 PermissionManagement 模組提供使用者資訊。Dashboard 的快速操作連結可能指向其他模組（如 FileManager、PermissionManagement 等）。

#### PermissionManagement（權限管理）模組

**本專案的需求背景：**

企業需要一個完整的權限管理系統來管理使用者、角色、組織與權限。傳統的簡單權限控制無法滿足複雜的組織架構與權限需求。系統需要支援：

1. **多層級組織架構：** 支援組織階層關係，可以建立多層級的組織架構
2. **角色權限管理：** 支援角色的導航權限設定，可以控制角色可以存取哪些功能
3. **使用者角色歸屬：** 使用者可以屬於多個角色，角色可以屬於多個組織
4. **組織圖視覺化：** 提供組織圖的視覺化顯示，幫助管理者了解組織架構

**主要功能：**

- **功能一：** 使用者管理 - 查詢、新增、修改、刪除使用者，設定使用者的角色與組織歸屬
- **功能二：** 角色管理 - 查詢、新增、修改、刪除角色，設定角色的導航權限
- **功能三：** 組織管理 - 查詢、新增、修改、刪除組織，建立組織階層關係
- **功能四：** 組織圖顯示 - 視覺化顯示組織架構，支援展開/收合、拖曳等功能

**查詢資料來源：**

- HnbHnbBackoffice.permission_management：權限管理主檔
- HnbHnbBackoffice.vw_permission_user：使用者權限檢視表
- HnbHnbBackoffice.vw_permission_role：角色權限檢視表
- HnbHnbBackoffice.vw_permission_organization：組織權限檢視表

**專案中這些資料的用途：**

- permission_management：統一儲存使用者、角色、組織的資料與權限
- vw_permission_user：提供使用者的整合查詢介面，包含使用者的組織資訊
- vw_permission_role：提供角色的整合查詢介面，包含角色的權限資訊
- vw_permission_organization：提供組織的整合查詢介面，包含組織的成員資訊

**異動資料來源：**

- HnbHnbBackoffice.permission_management：新增、修改、刪除使用者、角色、組織的資料

**本專案的操作邏輯：**

1. **使用者管理：**
   - 查詢：根據組織範圍（OrganizationScope）過濾使用者清單，只顯示當前使用者有權限查看的組織的使用者
   - 新增/修改：驗證必填欄位（使用者名稱、密碼、組織等），設定使用者的角色歸屬（多對多關係）
   - 刪除：檢查使用者是否有其他資料關聯（如檔案擁有者、團隊成員等），確認後刪除

2. **角色管理：**
   - 查詢：根據組織範圍過濾角色清單
   - 新增/修改：驗證必填欄位，設定角色的導航權限（可能為字串陣列或 JSON）
   - 刪除：檢查角色是否有使用者關聯，確認後刪除

3. **組織管理：**
   - 查詢：根據組織範圍過濾組織清單，支援組織階層關係的查詢
   - 新增/修改：驗證必填欄位，設定組織的上層組織（parent_id），建立組織階層關係
   - 刪除：檢查組織是否有下層組織或成員，確認後刪除

4. **組織圖顯示：**
   - 查詢：遞迴查詢組織階層關係，建立樹狀結構
   - 顯示：使用視覺化工具（如 Canvas、D3.js 等）顯示組織圖，支援展開/收合、拖曳等功能

**專案特定的規則：**

- **組織範圍（OrganizationScope）：** 使用者只能管理自己組織範圍內的使用者、角色、組織。這是權限控制的核心規則，確保不同組織的資料隔離。

**規則原因：**

企業通常有多個組織（部門、分公司等），不同組織的資料應該隔離。OrganizationScope 確保使用者只能管理自己組織範圍內的資料，提升資料安全性。

**本專案中模組間的關聯：**

PermissionManagement 模組是系統的核心模組，其他模組都依賴此模組進行權限控制：

- **Authorize 模組：** 使用 PermissionManagement 驗證使用者身份
- **BaseController：** 使用 PermissionAttribute 檢查使用者權限
- **SidebarNavigation 模組：** 根據使用者的角色權限顯示不同的導航項目
- **FileManager 模組：** 使用 PermissionManagement 檢查檔案的存取權限

#### FileManager（檔案管理）模組

**本專案的需求背景：**

企業需要一個檔案管理系統來管理檔案的上傳、下載、分享、預覽等。傳統的檔案伺服器方式缺乏權限控制與管理功能。系統需要支援：

1. **多使用者檔案隔離：** 每個使用者有自己的檔案儲存空間，檔案之間隔離
2. **檔案分享功能：** 使用者可以將檔案分享給其他使用者
3. **檔案預覽功能：** 支援多種檔案類型的預覽（圖片、PDF、文字檔等）
4. **資料夾管理：** 支援資料夾的新增、刪除、重新命名等

**主要功能：**

- **功能一：** 檔案上傳 - 支援單檔案與多檔案上傳，支援資料夾結構上傳
- **功能二：** 檔案查詢 - 根據路徑查詢檔案清單，支援多種檢視模式（清單、網格等）
- **功能三：** 檔案下載 - 根據權限檢查後提供檔案下載
- **功能四：** 檔案預覽 - 支援圖片、PDF、文字檔等類型的預覽
- **功能五：** 檔案分享 - 設定檔案的分享對象，允許其他使用者存取
- **功能六：** 資料夾管理 - 新增、刪除、重新命名資料夾

**查詢資料來源：**

- HnbHnbBackoffice.file_manager、vw_file_manager：檔案元資料
- 檔案系統（Areas/Backoffice/storage）：實際檔案

**專案中這些資料的用途：**

- file_manager：儲存檔案的元資料（名稱、路徑、大小、擁有者等），用於查詢、排序、權限控制
- 檔案系統：儲存實際檔案，提供檔案存取

**異動資料來源：**

- HnbHnbBackoffice.file_manager：新增、修改、刪除檔案元資料
- 檔案系統：新增、刪除實際檔案

**本專案的操作邏輯：**

1. **檔案上傳：**
   - 接收上傳的檔案，驗證檔案大小與類型
   - 儲存實際檔案到使用者的儲存目錄（Areas/Backoffice/storage/{username}/...）
   - 建立檔案元資料記錄到 file_manager，設定擁有者為當前使用者
   - 支援資料夾結構上傳，自動建立對應的資料夾

2. **檔案查詢：**
   - 根據路徑查詢檔案清單，過濾當前使用者有權限查看的檔案（擁有者或分享對象）
   - 支援多種檢視模式（清單、網格、詳細資料等）
   - 支援檔案類型篩選、排序等功能

3. **檔案下載：**
   - 檢查使用者是否有權限存取此檔案（擁有者或分享對象）
   - 如果無權限，返回 403 Forbidden
   - 如果有權限，從檔案系統讀取檔案並返回給使用者

4. **檔案分享：**
   - 檢查使用者是否為檔案擁有者
   - 設定檔案的分享對象（shared_with 欄位），允許其他使用者存取
   - 更新檔案元資料

5. **資料夾管理：**
   - 新增資料夾：在檔案系統中建立資料夾，並在資料庫中建立對應的記錄
   - 刪除資料夾：刪除資料夾及其所有內容（遞迴刪除），並刪除資料庫記錄
   - 重新命名資料夾：更新檔案系統中的資料夾名稱與資料庫記錄

**專案特定的規則：**

- **使用者儲存隔離：** 每個使用者有自己的儲存目錄（Areas/Backoffice/storage/{username}/），檔案之間隔離。這是為了確保不同使用者的檔案不會互相干擾。
- **檔案分享權限：** 只有檔案擁有者可以設定檔案的分享對象。分享對象可以存取檔案，但無法修改或刪除檔案（除非擁有者明確授權）。
- **虛擬路徑與實際路徑：** 使用者看到的是虛擬路徑（如 `/MyFiles/...`），實際檔案儲存在使用者的儲存目錄中。這樣的設計可以隱藏實際檔案系統結構，提供更友善的使用者介面。

**規則原因：**

虛擬路徑提供更友善的使用者體驗，使用者不需要了解實際檔案系統結構。實際路徑隱藏在系統內部，提升安全性。

**本專案中模組間的關聯：**

FileManager 模組依賴 PermissionManagement 模組進行權限控制，檢查檔案的存取權限。

**規則原因：**

硬體監控需要持續收集資料，才能及時發現系統問題。自動化的監控機制可以確保資料的完整性與即時性。

## 7. 重要功能詳解

### 7.1 權限管理系統

#### 7.1.1 功能設計背景

**需要解決的問題：**

企業需要一個完整的權限管理系統來管理使用者、角色、組織與權限。傳統的簡單權限控制無法滿足複雜的組織架構與權限需求。系統需要支援多層級組織架構、角色權限管理、使用者角色歸屬、組織圖視覺化等功能。

**問題的複雜度：**

權限管理涉及多個面向：

1. **多層級組織架構：** 支援組織階層關係，可以建立多層級的組織架構，查詢時需要遞迴查詢或使用檢視表
2. **角色權限管理：** 支援角色的導航權限設定，可以控制角色可以存取哪些功能，權限以字串陣列或 JSON 儲存
3. **使用者角色歸屬：** 使用者可以屬於多個角色，角色可以屬於多個組織，這是多對多的關係
4. **組織範圍控制：** 使用者只能管理自己組織範圍內的資料，需要實現組織範圍過濾（OrganizationScope）

#### 7.1.2 設計決策的思考過程

**設計目標：**

建立一個統一且靈活的權限管理系統，可以支援複雜的組織架構與權限需求。系統應該提供清晰的權限控制邏輯，易於維護與擴展。

**設計方案比較：**

| 方案 | 優點 | 缺點 | 適用場景 | 為什麼不採用 |
|------|------|------|----------|-------------|
| 統一資料表（permission_management） | • 統一管理<br>• 邏輯簡單<br>• 易於維護 | • 資料表結構較複雜<br>• 需要 type 欄位區分 | 需要統一管理不同類型的權限實體 | - |
| 分離資料表（users、roles、organizations） | • 結構清晰<br>• 符合正規化原則 | • 邏輯複雜<br>• 維護成本高<br>• 權限控制困難 | 不同類型實體差異較大 | 本專案需要統一管理權限，分離資料表會增加複雜度 |

**最終設計的考量：**

| 設計決策點 | 選擇的方案 | 為什麼這樣選擇 | 犧牲了什麼 | 如何權衡 |
|-----------|----------|--------------|-----------|----------|
| 資料表設計 | 統一使用 permission_management，使用 type 欄位區分 | 統一管理使用者、角色、組織，簡化權限控制邏輯 | 資料表結構較複雜，需要 type 欄位判斷 | 統一管理可以簡化權限控制，提升維護性 |
| 組織階層關係 | 使用 parent_id 建立自我關聯 | 支援多層級組織架構，查詢時可以使用遞迴或檢視表 | 查詢需要遞迴，效能可能受影響 | 組織階層關係對業務很重要，可以透過檢視表優化查詢 |
| 角色權限儲存 | 使用字串陣列或 JSON 儲存導航權限 | 提供靈活的權限設定，支援動態權限調整 | 無法使用資料庫外鍵約束，需要在應用程式層面驗證 | 靈活性比外鍵約束更重要，可以在應用程式層面驗證 |
| 組織範圍控制 | 使用 OrganizationScope 過濾資料 | 確保使用者只能管理自己組織範圍內的資料，提升資料安全性 | 查詢時需要額外過濾，可能影響效能 | 資料安全性比效能更重要，可以透過索引優化查詢 |

**從這個設計中學到的模式：**

當設計權限管理系統時，應該考慮：1. 權限實體的統一性與分離性的權衡 2. 組織階層關係的儲存與查詢方式 3. 權限的儲存方式（外鍵關聯 vs. 字串陣列）4. 組織範圍控制的實現方式。統一管理可以簡化邏輯，但需要妥善設計資料表結構。組織階層關係可以使用自我關聯，但需要考慮查詢效能。

#### 7.1.3 功能運作原理

**核心邏輯的思考過程：**

權限管理系統的核心是統一管理使用者、角色、組織，並根據使用者的組織範圍控制資料存取。系統採用以下邏輯：

1. **統一資料模型：** 使用 permission_management 統一儲存使用者、角色、組織的資料，使用 type 欄位區分
2. **組織階層關係：** 使用 parent_id 建立組織階層關係，形成樹狀結構
3. **權限控制：** 使用 PermissionAttribute 檢查使用者權限，使用 OrganizationScope 過濾資料範圍
4. **檢視表輔助：** 使用 vw_permission_user、vw_permission_role、vw_permission_organization 提供特定類型的查詢介面

**運作流程：**

| 步驟 | 說明 | 設計考量 | 可能的替代方案 |
|------|------|----------|---------------|
| 1. 使用者登入 | 驗證使用者身份，載入使用者資訊 | 使用 Cookie 驗證，載入使用者資訊到 Claims | 使用 JWT，但 Cookie 更適合 Web 應用 |
| 2. 載入使用者權限 | 根據使用者的角色載入權限資訊 | 查詢使用者的角色，載入角色的導航權限 | 每次都查詢資料庫，但可以快取權限資訊 |
| 3. 檢查權限 | 使用 PermissionAttribute 檢查使用者是否有權限存取功能 | 在 Controller 層檢查權限，未授權時返回 AccessDenied | 在 Service 層檢查權限，但 Controller 層檢查更早，效能更好 |
| 4. 過濾資料範圍 | 使用 OrganizationScope 過濾資料，只顯示使用者組織範圍內的資料 | 在 Service 層過濾資料，確保資料安全性 | 在 Repository 層過濾資料，但 Service 層過濾更靈活 |

**設計模式與原則的應用：**

- **Repository Pattern：** 使用 Repository 層封裝資料存取邏輯，降低耦合度
- **Service Layer Pattern：** 使用 Service 層處理業務邏輯，保持 Controller 的簡潔
- **Attribute Pattern：** 使用 PermissionAttribute 統一檢查權限，降低程式碼重複
- **Scope Pattern：** 使用 OrganizationScope 統一過濾資料範圍，確保資料安全性

### 7.2 檔案管理系統

#### 7.2.1 功能設計背景

**需要解決的問題：**

企業需要一個檔案管理系統來管理檔案的上傳、下載、分享、預覽等。傳統的檔案伺服器方式缺乏權限控制與管理功能。系統需要支援多使用者檔案隔離、檔案分享功能、檔案預覽功能、資料夾管理等。

**問題的複雜度：**

檔案管理涉及多個面向：

1. **多使用者檔案隔離：** 每個使用者有自己的檔案儲存空間，檔案之間隔離，需要確保使用者無法存取其他使用者的檔案
2. **檔案分享功能：** 使用者可以將檔案分享給其他使用者，需要管理分享對象與權限
3. **檔案元資料管理：** 檔案元資料儲存在資料庫，實際檔案儲存在檔案系統，需要確保兩者的一致性
4. **權限控制：** 檔案的存取權限需要根據擁有者與分享對象判斷

#### 7.2.2 設計決策的思考過程

**設計目標：**

建立一個安全且靈活的檔案管理系統，支援多使用者隔離、檔案分享、權限控制等功能。系統應該提供良好的使用者體驗，同時確保資料安全性。

**設計方案比較：**

| 方案 | 優點 | 缺點 | 適用場景 | 為什麼不採用 |
|------|------|------|----------|-------------|
| 混合儲存（元資料在資料庫，檔案在檔案系統） | • 利用資料庫查詢能力<br>• 避免資料庫儲存大檔案<br>• 支援檔案分享與權限控制 | • 需要確保兩者一致性<br>• 備份較複雜 | 需要查詢與權限控制功能 | - |
| 全部儲存在資料庫 | • 一致性保證<br>• 備份簡單 | • 資料庫體積大<br>• 查詢效能差<br>• 不適合大檔案 | 檔案較小，不需要複雜查詢 | 本專案檔案可能較大，且需要複雜查詢與權限控制 |
| 全部儲存在檔案系統 | • 簡單直接<br>• 效能好 | • 無法查詢<br>• 無法權限控制<br>• 難以管理 | 檔案較少，不需要查詢與權限控制 | 本專案需要查詢與權限控制功能 |

**最終設計的考量：**

| 設計決策點 | 選擇的方案 | 為什麼這樣選擇 | 犧牲了什麼 | 如何權衡 |
|-----------|----------|--------------|-----------|----------|
| 儲存方式 | 元資料在資料庫，檔案在檔案系統 | 利用資料庫查詢能力，避免資料庫儲存大檔案 | 需要確保兩者一致性，備份較複雜 | 查詢與權限控制比一致性更重要，可以透過程式邏輯確保一致性 |
| 使用者隔離 | 每個使用者有自己的儲存目錄 | 確保不同使用者的檔案隔離，提升安全性 | 路徑管理較複雜 | 資料安全性比路徑管理複雜度更重要 |
| 虛擬路徑 | 使用者看到虛擬路徑，實際路徑隱藏 | 提供更友善的使用者體驗，隱藏實際檔案系統結構 | 需要路徑轉換邏輯 | 使用者體驗比路徑轉換邏輯更重要 |

**從這個設計中學到的模式：**

當設計檔案管理系統時，應該考慮：1. 儲存方式的選擇（資料庫 vs. 檔案系統 vs. 混合）2. 使用者隔離的實現方式 3. 檔案分享與權限控制的設計 4. 檔案元資料與實際檔案的一致性保證。混合儲存可以結合資料庫與檔案系統的優點，但需要確保兩者一致性。

### 7.3 AI 功能整合

4. **模組化設計：** AI 功能應該封裝為獨立模組，可以獨立測試與維護

#### 7.3.2 設計決策的思考過程

**設計目標：**

建立一個模組化且可擴展的 AI 功能整合架構，支援多種 AI 功能，同時確保系統的可維護性與可測試性。

| 外部服務（微服務） | • 完全獨立<br>• 可獨立部署 | • 複雜度高<br>• 需要服務間通訊 | 大型分散式系統 | 專案規模尚未達到微服務的需求，模組化設計已足夠 |

**最終設計的考量：**

| 設計決策點 | 選擇的方案 | 為什麼這樣選擇 | 犧牲了什麼 | 如何權衡 |
|-----------|----------|--------------|-----------|----------|
| 模組組織 | 使用 IntelligentSystems 目錄組織 AI 模組 | 模組化設計，每個模組獨立，易於測試與維護 | 需要模組間整合 | 模組化設計可以提升可維護性與可擴展性 |
| 模型管理 | 使用 ModelHealthChecker 管理模型健康狀態 | 自動檢查模型是否存在，自動下載，確保 AI 功能可用 | 啟動時需要下載模型，可能影響啟動時間 | AI 功能的可用性比啟動時間更重要 |
| API 整合 | 使用 IHttpClientFactory 管理 HTTP 連線 | 提供連線池管理、重試機制等，提升可靠性 | 需要額外的設定 | HTTP 連線管理對 API 整合很重要 |
| 配置管理 | 使用 appsettings.json 管理配置 | 集中管理配置，易於調整 | 敏感資訊可能暴露 | 可以使用環境變數或秘密管理工具管理敏感資訊 |

**從這個設計中學到的模式：**

## 8. 資料處理規則

### 8.1 權限資料驗證規則

#### 規則設計的問題識別

**需要解決的問題：**

權限資料的正確性與完整性是系統運作的基礎。如果權限資料不正確或不完整，會影響使用者的存取權限，導致安全問題或功能無法使用。系統需要確保所有權限資料都符合業務規則與資料格式要求。

**問題的複雜度：**

權限資料驗證涉及多個面向：

1. **必填欄位檢查：** 不同類型的權限實體（使用者、角色、組織）有不同的必填欄位要求
2. **唯一性驗證：** 使用者名稱、角色名稱、組織名稱等欄位需要確保唯一性
3. **組織階層關係驗證：** 組織的 parent_id 必須指向存在的組織，且不能形成循環引用
4. **角色權限格式驗證：** 角色的導航權限以字串陣列或 JSON 儲存，需要驗證格式正確性

#### 規則設計的思考過程

**設計目標：**

確保所有權限資料在儲存前都經過完整的驗證，包括必填欄位、唯一性、組織階層關係等。驗證失敗時應該返回明確的錯誤訊息，協助使用者修正輸入。

**最終設計的考量：**

| 驗證類型 | 判斷方式 | 設計原因 | 判斷方式的選擇 | 其他可能的判斷方式 |
|----------|----------|----------|---------------|------------------|
| 必填欄位 | 檢查欄位是否為空或 null | 缺失必填欄位會導致資料不完整 | 在 Repository 層檢查，統一處理 | 使用 Data Annotations，但無法根據類型動態檢查 |
| 唯一性驗證 | 查詢既有資料，檢查是否有重複 | 確保使用者名稱、角色名稱、組織名稱等唯一 | 在 Repository 層查詢，確保資料一致性 | 使用資料庫唯一約束，但應用程式層面驗證可以提供更清楚的錯誤訊息 |
| 組織階層關係 | 檢查 parent_id 是否指向存在的組織，檢查是否形成循環引用 | 組織階層關係錯誤會導致查詢失敗或邏輯錯誤 | 在 Repository 層遞迴檢查，確保階層關係正確 | 使用資料庫觸發器，但應用程式層面驗證可以提供更清楚的錯誤訊息 |

**從權限驗證設計中學到的模式：**

權限資料驗證應該：1. 在 Repository 層進行驗證，確保所有資料存取都經過驗證 2. 驗證應該考慮不同類型的權限實體（使用者、角色、組織）有不同的驗證規則 3. 驗證失敗時應該返回明確的錯誤訊息，協助使用者修正輸入 4. 組織階層關係驗證需要防止循環引用，確保階層關係的正確性。

### 8.2 檔案元資料與實際檔案的一致性規則

#### 規則設計的問題識別

**需要解決的問題：**

檔案管理的元資料儲存在資料庫（file_manager），實際檔案儲存在檔案系統。如果兩者不一致，會導致查詢到不存在的檔案或檔案系統中存在但資料庫中沒有記錄的檔案，影響系統的正確性與效能。

**問題的複雜度：**

檔案一致性涉及多個面向：

1. **新增檔案時的一致性：** 檔案上傳時需要同時在資料庫與檔案系統中建立記錄
2. **刪除檔案時的一致性：** 檔案刪除時需要同時刪除資料庫記錄與檔案系統中的檔案
3. **重新命名檔案時的一致性：** 檔案重新命名時需要同時更新資料庫記錄與檔案系統中的檔案名稱
4. **檔案移動時的一致性：** 檔案移動時需要同時更新資料庫記錄與檔案系統中的檔案位置

#### 更新邏輯的設計思考

**更新邏輯的設計目標：**

確保檔案元資料與實際檔案始終保持一致，避免不一致造成的問題。更新邏輯應該原子化，確保要麼全部成功，要麼全部失敗。

**更新邏輯的設計方案比較：**

| 方案 | 優點 | 缺點 | 適用場景 | 為什麼不採用 |
|------|------|------|----------|-------------|
| 先更新資料庫，後更新檔案系統 | • 資料庫更新快速<br>• 可以立即查詢 | • 檔案系統操作失敗會導致不一致<br>• 需要回滾機制 | 檔案系統操作較可靠 | 檔案系統操作可能失敗，需要回滾機制 |
| 先更新檔案系統，後更新資料庫 | • 檔案系統操作失敗不會影響資料庫 | • 資料庫更新失敗會導致檔案系統中存在但資料庫中沒有記錄的檔案 | 檔案系統操作較不可靠 | 資料庫更新失敗會導致孤兒檔案 |
| 使用交易機制 | • 可以確保原子性<br>• 失敗時自動回滾 | • 檔案系統操作無法納入資料庫交易<br>• 需要額外的補償機制 | 需要強一致性保證 | 檔案系統操作無法納入資料庫交易，無法實現真正的原子性 |

**最終設計的考量：**

系統採用先更新資料庫，後更新檔案系統的方式，如果檔案系統操作失敗，使用補償機制刪除資料庫記錄。這是因為：

1. **資料庫更新快速：** 資料庫更新比檔案系統操作快速，可以立即反映變更
2. **補償機制：** 檔案系統操作失敗時，使用補償機制刪除資料庫記錄，確保一致性
3. **錯誤處理：** 檔案系統操作失敗時返回錯誤訊息，使用者可以重試

**從檔案一致性設計中學到的模式：**

檔案一致性管理應該：1. 採用先更新資料庫，後更新檔案系統的方式，確保資料庫始終反映最新狀態 2. 檔案系統操作失敗時使用補償機制，確保一致性 3. 定期檢查檔案一致性，發現並修復不一致的問題 4. 提供檔案一致性檢查工具，協助管理者維護系統。

## 9. 故障排除與常見問題

### 9.1 錯誤日誌位置

| 錯誤類型 | 記錄位置 | 識別方式 | 說明 |
|----------|----------|----------|------|
| 系統例外錯誤 | Hnbdata.error_logs | `EventName` 欄位包含功能名稱或例外類型 | 系統運行過程中的例外錯誤，記錄詳細的堆疊追蹤資訊 |

| 封鎖 IP 記錄 | Hnbdata.blocked_ips | `IpAddress`、`CreatedDate` 等欄位 | 記錄被封鎖的 IP 地址，用於 IP 安全控制 |
| AI 功能日誌 | HnbHnbBackoffice.ai_log、vw_ai_log | `CreatedDate`、`EventName` 等欄位 | 記錄 AI 功能的執行記錄，包含 API 呼叫、模型載入等 |

**錯誤日誌關鍵字/識別對照：**

| 關鍵字/識別碼 | 模組/功能 | 說明 |
|--------------|----------|------|
| `Backoffice/Dashboard` | Dashboard 模組 | 儀表板相關錯誤 |
| `Backoffice/Authorize` | Authorize 模組 | 登入/授權相關錯誤 |
| `Backoffice/PermissionManagement` | PermissionManagement 模組 | 權限管理相關錯誤 |
| `Backoffice/FileManager` | FileManager 模組 | 檔案管理相關錯誤 |
| `ModelHealthChecker` | IntelligentSystems 模組 | 模型健康檢查相關錯誤 |

**本專案的錯誤日誌設計理念：**

本專案將錯誤日誌分散記錄到不同的資料表（error_logs、access_records、blocked_ips、ai_log 等），使用 `EventName` 或功能名稱標識錯誤來源。這樣的設計使得錯誤日誌分類清晰，方便查詢與分析。每個模組使用不同的錯誤標識，例如 Dashboard 使用 `Backoffice/Dashboard`。錯誤日誌記錄包含 `CreatedDate`、`CreatedUserName`、詳細錯誤訊息等欄位，提供完整的錯誤資訊。

### 9.2 錯誤日誌查詢方式

1. **系統例外錯誤（Hnbdata.error_logs）：**
   - 可依 `EventName`、`CreatedDate`、`CreatedUserName` 等條件篩選
   - 用於查詢特定功能或時間範圍的錯誤記錄
   - **專案中的查詢技巧：** 使用 `EventName` 欄位篩選特定模組的錯誤（例如 `Backoffice/FileManager`），使用 `CreatedDate` 篩選特定時間範圍的錯誤，使用 `CreatedUserName` 篩選特定使用者的錯誤。錯誤訊息中通常包含詳細的堆疊追蹤資訊，可以快速定位問題程式碼。

2. **AI 功能日誌（HnbHnbBackoffice.ai_log、vw_ai_log）：**
   - 使用 `EventName`、`CreatedDate` 等欄位篩選
   - 用於查詢 AI 功能的執行記錄，包含 API 呼叫、模型載入等
   - **專案中的查詢技巧：** 使用 `EventName` 篩選特定 AI 功能的記錄（例如物件辨識、圖像生成），使用 `CreatedDate` 篩選特定時間範圍的記錄。日誌中通常包含 API 回應、模型狀態等資訊，可以幫助診斷 AI 功能問題。

3. **硬體監控資料（HnbHnbBackoffice.hardware_monitoring、vw_hardware_monitoring）：**
   - 使用 `CreatedDate`、`is_active` 等欄位篩選
   - 用於查詢硬體監控資料，分析系統效能問題
   - **專案中的查詢技巧：** 使用 `CreatedDate` 篩選特定時間範圍的監控資料，使用 `is_active` 篩選啟用的監控項目。監控資料包含 CPU、GPU、記憶體等使用率，可以幫助診斷效能問題。

### 9.3 錯誤定位的思考方法

**錯誤排查的思維框架：**

錯誤排查應該遵循從外到內的思維框架：1. 先確認錯誤的徵兆與範圍（是特定功能、特定使用者、特定時間等）2. 再查看錯誤日誌，了解錯誤的詳細資訊 3. 最後追蹤到具體的程式碼位置，分析錯誤的根本原因。這樣的思維框架可以快速定位問題，避免盲目的排查。

**錯誤排查流程的設計：**

1. **前端/使用者介面錯誤：**
   - **為什麼先檢查這裡：** 使用者最直接感受到的是前端錯誤，例如頁面無法載入、按鈕無反應等。前端錯誤通常是後端問題的表現，但前端錯誤的徵兆最明顯，可以先從前端開始排查
   - **如何快速判斷：** 檢查瀏覽器的開發者工具（F12），查看 Console 是否有 JavaScript 錯誤、Network 是否有請求失敗、Response 是否有錯誤訊息
   - **常見的誤判情況：** 前端顯示的錯誤可能是後端錯誤的表現，例如後端回傳錯誤 JSON，前端顯示錯誤訊息。應該檢查 Network 中的 Response，確認錯誤的真正來源
   - **排查效率的考量：** 前端錯誤的徵兆明顯，可以快速判斷問題是否在前端，但需要注意前端錯誤可能是後端問題的表現

2. **後端/Controller 層錯誤：**
   - **為什麼需要檢查這裡：** Controller 層是請求處理的第一個點，負責輸入驗證、流程協調等。如果 Controller 層有問題，例如參數驗證失敗、路由錯誤等，會導致請求無法正確處理
   - **如何快速判斷：** 檢查 Controller 的日誌記錄，查看是否有例外錯誤。使用 ASP.NET Core 的開發環境，可以查看詳細錯誤頁面，了解例外錯誤的詳細資訊
   - **常見的誤判情況：** Controller 層的錯誤可能是 Service 層或 Repository 層錯誤的表現，例如 Service 層拋出例外，Controller 層記錄錯誤。應該追蹤到 Service 層，了解錯誤的根本原因
   - **排查效率的考量：** Controller 層的錯誤通常有清楚的錯誤訊息，可以快速定位問題，但需要追蹤到 Service 層或 Repository 層，了解錯誤的根本原因

3. **資料庫/資料層錯誤：**
   - **為什麼最後檢查這裡：** 資料庫錯誤通常是應用程式錯誤的最終表現，例如查詢失敗、資料不存在、連線失敗等。資料庫錯誤的排查需要檢查連線字串、資料表結構、資料內容等，相對複雜
   - **如何快速判斷：** 檢查資料庫連線是否正常，檢查 SQL 查詢是否正確，檢查資料表是否存在，檢查資料是否存在。使用 PostgreSQL 管理工具直接查詢資料庫，確認資料狀態
   - **常見的誤判情況：** 資料庫錯誤可能是應用程式邏輯錯誤的表現，例如查詢條件錯誤導致查詢失敗。應該檢查應用程式的查詢邏輯，確認查詢條件是否正確
   - **排查效率的考量：** 資料庫錯誤的排查需要檢查多個層面，相對複雜，但資料庫錯誤通常是問題的最終表現，排查到資料庫層面可以找到問題的根本原因

**系統化排查的思維模式：**

系統化排查應該遵循：1. 先確認錯誤的徵兆與範圍 2. 再查看錯誤日誌，了解錯誤的詳細資訊 3. 追蹤到具體的程式碼位置，分析錯誤的根本原因 4. 驗證修復方案，確認問題已解決。這個思維模式可以應用到任何錯誤排查場景，確保排查過程系統化且高效。

### 9.4 常見問題的思考與解決

#### Q1：使用者無法登入系統

**問題的識別：**

使用者嘗試登入系統時，系統回傳「使用者名稱或密碼錯誤」或「帳號已被停用」等錯誤訊息。問題的徵兆是登入失敗，使用者無法存取系統。

**問題的根本原因分析：**

登入功能需要驗證使用者身份，檢查使用者是否存在、密碼是否正確、帳號是否啟用等。可能的原因包括：

1. **使用者名稱或密碼錯誤：** 使用者輸入的使用者名稱或密碼不正確
2. **帳號已被停用：** 使用者的帳號被停用，無法登入
3. **使用者不存在：** 資料庫中找不到該使用者
4. **資料庫連線失敗：** 無法連線到資料庫，導致無法查詢使用者資料
5. **Cookie 設定錯誤：** Cookie 設定錯誤，導致登入狀態無法保存

**解決方案的思考：**

- **方案一：** 在錯誤訊息中明確說明錯誤原因（使用者名稱錯誤、密碼錯誤、帳號停用等）
  - **優缺點：** 優點是可以幫助使用者快速了解問題，缺點是可能暴露系統資訊（例如使用者是否存在）
  - **為什麼選擇：** 這個方案可以幫助使用者快速了解問題並解決，雖然可能暴露系統資訊，但可以透過模糊化錯誤訊息來降低風險

- **方案二：** 統一錯誤訊息「使用者名稱或密碼錯誤」，不區分具體原因
  - **優缺點：** 優點是不會暴露系統資訊，缺點是使用者無法快速了解問題
  - **為什麼不選擇：** 這個方案雖然安全性較高，但會降低使用者體驗

**最終解決方案：**

在錯誤訊息中明確說明錯誤原因，但對於安全性敏感的資訊（例如使用者是否存在）使用模糊化的錯誤訊息。例如：使用者名稱錯誤時顯示「使用者名稱或密碼錯誤」，帳號停用時顯示「帳號已被停用，請聯絡管理員」。

**問題的預防思考：**

1. **提供密碼重設功能：** 如果使用者忘記密碼，可以透過密碼重設功能重新設定密碼
2. **帳號啟用提醒：** 帳號被停用時，系統可以發送通知提醒使用者或管理員
3. **登入嘗試限制：** 限制登入嘗試次數，防止暴力破解

**從這個問題中學到的經驗：**

1. **錯誤訊息應該明確但不暴露敏感資訊：** 錯誤訊息應該幫助使用者快速了解問題，但不應該暴露系統的敏感資訊（例如使用者是否存在）
2. **提供自助解決方案：** 對於常見問題（例如忘記密碼），應該提供自助解決方案，減少管理員的負擔
3. **安全性與使用者體驗的權衡：** 在設計錯誤訊息時，需要在安全性與使用者體驗之間權衡，選擇最適合的方案

#### Q2：AI 功能無法運作，提示「模型檔案不存在」

**問題的識別：**

使用者嘗試使用 AI 功能（例如物件辨識）時，系統回傳「模型檔案不存在」或「模型初始化失敗」等錯誤訊息。問題的徵兆是 AI 功能無法使用。

**問題的根本原因分析：**

AI 功能需要載入 ONNX 模型檔案才能運作。ModelHealthChecker 會在啟動時檢查模型是否存在，如果不存在會自動下載。可能的原因包括：

1. **模型檔案路徑錯誤：** `appsettings.json` 中的模型路徑設定錯誤，導致系統找不到模型檔案
2. **模型檔案下載失敗：** 模型檔案下載時網路連線失敗，導致下載失敗
3. **模型檔案損壞：** 模型檔案下載完成後損壞，導致無法載入
4. **模型檔案權限不足：** 模型檔案所在目錄的權限不足，導致無法讀取模型檔案
5. **ModelHealthChecker 未啟動：** ModelHealthChecker 未正常啟動，導致模型檢查與下載機制未運作

**解決方案的思考：**

- **方案一：** 在 ModelHealthChecker 中加強錯誤處理，提供詳細的錯誤訊息與修復建議
  - **優缺點：** 優點是可以幫助使用者快速了解問題並解決，缺點是增加了系統複雜度
  - **為什麼選擇：** 這個方案可以幫助使用者快速了解問題並解決，提升使用者體驗

- **方案二：** 提供模型檔案手動上傳功能，允許使用者手動上傳模型檔案
  - **優缺點：** 優點是提供了備援方案，缺點是需要額外的檔案上傳功能
  - **為什麼選擇：** 這個方案提供了備援方案，當自動下載失敗時，使用者可以手動上傳模型檔案

**最終解決方案：**

1. 在 ModelHealthChecker 中加強錯誤處理，提供詳細的錯誤訊息與修復建議
2. 提供模型檔案手動上傳功能，允許使用者手動上傳模型檔案
3. 在錯誤日誌中記錄模型檢查與下載的詳細資訊，協助診斷問題

**問題的預防思考：**

1. **模型檔案備份：** 將模型檔案備份到多個位置，確保模型檔案的可用性
2. **模型檔案完整性檢查：** 下載模型檔案後檢查檔案完整性，確保檔案未損壞
3. **模型檔案路徑驗證：** 在啟動時驗證模型檔案路徑，確保路徑正確

**從這個問題中學到的經驗：**

1. **自動化機制需要有備援方案：** 自動化的機制（例如模型自動下載）可能失敗，需要有備援方案（例如手動上傳）
2. **錯誤處理應該提供詳細資訊：** 錯誤處理應該提供詳細的錯誤訊息與修復建議，幫助使用者快速解決問題
3. **日誌記錄的重要性：** 詳細的日誌記錄可以幫助診斷問題，應該在關鍵步驟記錄詳細資訊

## 10. 部署與維護

### 10.1 建置與部署流程

#### 10.1.1 建置

```bash
dotnet build HNB/Horizon_Nova/HNB/HNB.csproj
```

**本專案的建置考量：**

本專案使用 ASP.NET Core 8.0，需要在 .NET SDK 8.0 環境中建置。建置前需要確認：

1. **資料庫連線字串：** 確認 `appsettings.json` 中的資料庫連線字串正確
2. **AI 服務設定：** 確認 `appsettings.json` 中的 AI 服務設定（DALL-E 3 API Key、Qdrant 設定等）正確
3. **模型檔案路徑：** 確認 `appsettings.json` 中的模型檔案路徑設定正確
4. **儲存目錄權限：** 確認儲存目錄（`Areas/Backoffice/storage`）的權限設定正確，應用程式有讀寫權限

#### 10.1.2 發佈

```bash
dotnet publish HNB/Horizon_Nova/HNB/HNB.csproj -c Release -o ./publish
```

**本專案的發佈考量：**

發佈時需要注意：

1. **設定檔管理：** 發佈時需要區分開發環境與正式環境的設定檔（`appsettings.Development.json` vs. `appsettings.json`）
2. **敏感資訊管理：** 敏感資訊（API Key、密碼等）不應該直接寫在設定檔中，應該使用環境變數或秘密管理工具
3. **模型檔案處理：** 模型檔案較大，不適合包含在發佈包中，應該在部署後由 ModelHealthChecker 自動下載
4. **靜態檔案：** 確認靜態檔案（CSS、JS、圖片等）正確包含在發佈包中

#### 10.1.3 部署注意事項

| 項目 | 設定 | 說明 | 專案特定的設計原因 |

### 10.2 維護作業

#### 10.2.1 資料庫備份

**本專案中維護作業的目的：**

定期備份資料庫可以防止資料遺失，確保系統在發生問題時可以恢復。本專案使用兩個資料庫，需要分別備份。

**本專案中維護作業的背景：**

本專案使用 PostgreSQL 資料庫，可以使用 `pg_dump` 工具備份資料庫。備份應該包含兩個資料庫：Hnbdata 與 HnbHnbBackoffice。

| 步驟 | 操作 | 注意事項 | 專案特定的原因 |
|------|------|----------|---------------|
| 1 | 備份 Hnbdata 資料庫 | 使用 `pg_dump` 工具備份，儲存到安全位置 | 系統需要兩個資料庫，需要分別備份 |
| 2 | 備份 HnbHnbBackoffice 資料庫 | 使用 `pg_dump` 工具備份，儲存到安全位置 | 系統需要兩個資料庫，需要分別備份 |
| 3 | 備份檔案系統 | 備份 `Areas/Backoffice/storage` 目錄 | 檔案管理功能使用檔案系統儲存，需要備份檔案系統 |

**維護作業在專案中的影響：**

資料庫備份是維護作業，不會影響系統運作。但備份時可能會產生較大的檔案，需要確保儲存空間足夠。檔案系統備份時可能會影響檔案存取效能，建議在低峰時段執行。

#### 10.2.2 日誌清理

**本專案中維護作業的目的：**

定期清理舊日誌可以釋放資料庫空間，提升查詢效能。本專案的日誌分散在多個資料表，需要分別清理。

**本專案中維護作業的背景：**

本專案的日誌包括：error_logs、access_records、ai_log 等。這些日誌會隨著時間累積，需要定期清理。

| 步驟 | 操作 | 注意事項 | 專案特定的原因 |
|------|------|----------|---------------|
| 1 | 清理 error_logs | 刪除30天前的錯誤日誌，保留最近的日誌用於問題診斷 | 錯誤日誌用於問題診斷，保留最近的日誌即可 |
| 2 | 清理 access_records | 刪除30天前的存取記錄，保留最近的記錄用於安全審計 | 存取記錄用於安全審計，保留最近的記錄即可 |
| 3 | 清理 ai_log | 刪除30天前的 AI 功能日誌，保留最近的日誌用於問題診斷 | AI 功能日誌用於問題診斷，保留最近的日誌即可 |

**維護作業在專案中的影響：**

日誌清理會刪除資料，需要謹慎執行。建議先備份資料庫，確認備份成功後再執行清理。清理後需要檢查系統是否正常運作。

Horizon Nova 樹莓派遷移計畫書

一、設備確認與環境評估

1.1 樹莓派硬體規格清單

本專案採用 3 台樹莓派設備，分別用於不同的服務層級與功能角色：

主要運算設備：
- Raspberry Pi 3 Model B/B+：2 台
  - 用途：AI 運算節點，負責執行 AI 模型推理與處理任務
  - 處理器：Broadcom BCM2837，四核心 ARM Cortex-A53，時脈 1.2GHz
  - 記憶體：1GB LPDDR2（依型號而定）
  - 網路：10/100Mbps 有線網路，802.11n 無線網路

- Raspberry Pi 4 Model B：1 台
  - 用途：HomeLab 主控節點、中央監控系統、容器編排管理平台
  - 處理器：Broadcom BCM2711，四核心 ARM Cortex-A72，時脈 1.5GHz
  - 記憶體：4GB/8GB LPDDR4（依型號而定）
  - 網路：Gigabit 有線網路，802.11ac 無線網路，支援雙顯示輸出

散熱系統配置：
- 大型電腦風扇：1 台
  - 用途：整體機架散熱，確保疊放式架設環境下的空氣流通
  - 安裝位置：組合支架頂部或側面

- 小型 Pi 專用風扇：3 台
  - 用途：個別樹莓派晶片散熱，每台設備配備 1 台
  - 安裝方式：直接安裝於樹莓派 GPIO 針腳或專用散熱座

架設與結構設備：
- 組合式支架系統：1 組
  - 用途：將 3 台樹莓派以垂直疊放方式整合架設
  - 設計考量：提供穩固支撐、良好散熱空間、易於維護與線路管理
  - 結構特點：模組化設計，支援未來擴充需求

1.2 儲存空間評估與規劃

儲存媒介配置：
本專案採用樹莓派內建 SD 卡儲存方案，未使用外接儲存擴充設備。所有 3 台樹莓派均配置相同規格的儲存媒介。

SD 卡規格：
- 儲存容量：128GB microSD 卡（3 台設備統一規格）
- 儲存類型：Class 10 或以上等級，建議使用 UHS-I/UHS-II 規格以確保 I/O 效能
- 檔案系統：建議使用 ext4 檔案系統以獲得最佳效能與穩定性

空間分配規劃：
- 作業系統與基礎環境：預估佔用 10-20GB
  - Linux 發行版（Raspberry Pi OS 或 Ubuntu Server）
  - .NET 8.0 Runtime 與 SDK
  - 系統工具與依賴套件
  - 日誌檔案與暫存空間

- 應用程式與服務：預估佔用 5-10GB
  - Horizon Nova 應用程式本體
  - 容器映像檔（Docker images）
  - 應用程式日誌與快取

- 資料庫與資料儲存：預估佔用 10-30GB
  - PostgreSQL 資料庫檔案
  - 使用者上傳檔案
  - 系統產生的資料檔案

- 預留空間：建議保留 20-30GB 作為緩衝空間
  - 系統更新與升級
  - 臨時檔案與快取
  - 未來擴充需求

總可用空間評估：扣除系統佔用後，預估可用空間約 80-100GB，應足以支援初期營運需求。

1.3 網路環境確認與架構規劃

區域網路環境：
- 網路類型：家用區域網路（Local Area Network, LAN）
- 連線方式：有線網路（Ethernet）或無線網路（Wi-Fi）
- 網路架構：透過家用路由器連接至網際網路
- IP 分配：採用 DHCP 動態分配或靜態 IP 設定（建議使用靜態 IP 以確保服務穩定性）

外網存取解決方案：
本專案採用 Cloudflare Tunnel（原 Argo Tunnel）技術實現外網存取功能，此方案具有以下優勢：

- 無需申請固定 IP：不需要向網路服務供應商（ISP）申請固定 IP 位址
- 無需設定動態 DNS：不需要配置 DDNS 服務或路由器端口轉發
- 安全性提升：所有流量透過 Cloudflare 的加密通道傳輸，提供額外的安全層
- 成本效益：Cloudflare Tunnel 提供免費方案，無需額外費用
- 易於管理：透過 Cloudflare 控制台統一管理多個服務的對外存取

技術實作要求：
- 需要在樹莓派上安裝並運行 Cloudflare Tunnel 客戶端（cloudflared）
- 樹莓派需要持續運行以維持外網連線
- 需要 Cloudflare 帳號並完成網域設定

HomeLab 環境建置狀態：
目前尚未建立 HomeLab 環境，此為本遷移計畫的重要目標之一。HomeLab 環境將建置於 Raspberry Pi 4 上，作為整體基礎架構的核心，提供容器編排、服務管理、監控等基礎設施服務。

二、專案概述

2.1 專案背景與動機

現況分析：
目前 Horizon Nova 系統部署於 Railway 雲端平台，採用 Platform-as-a-Service（PaaS）模式運作。系統包含完整的後台管理功能、AI 功能整合、資料庫服務等核心模組。

成本問題：
根據目前的使用情況，Railway 平台每月產生約 20-25 美元的營運費用。此費用主要來自於：
- 運算資源租用費用
- 資料庫服務費用
- 網路流量費用
- 儲存空間費用

隨著系統使用量增加，預期成本將持續上升，長期營運成本考量下，需要尋求更具成本效益的解決方案。

遷移決策：
經過成本效益分析與技術可行性評估，決定將系統從雲端平台遷移至本地架設的樹莓派環境。此決策基於以下考量：

- 成本控制：樹莓派硬體為一次性投資，長期營運成本遠低於雲端服務
- 技術學習：透過本地架設過程，深入理解系統架構與運作機制
- 自主控制：完全掌控硬體與軟體環境，不受雲端平台限制
- 擴展性：可依需求逐步擴充硬體設備，靈活調整系統規模

2.2 遷移目標與技術架構規劃

整體目標：
本遷移計畫旨在建立一個完整的本地化運算環境，不僅僅是單純的系統遷移，更是一個完整的基礎設施建置專案。目標包括：

1. 建立穩定的本地運算環境
2. 實現成本效益優化
3. 建立可擴展的基礎架構
4. 整合 AI 運算能力
5. 建立完善的監控與管理機制

技術架構規劃：

外網存取層：
- 技術方案：Cloudflare Tunnel
- 功能角色：作為外部存取入口，提供安全的外網連線通道
- 實作內容：
  - Cloudflare 帳號設定與網域配置
  - Cloudflare Tunnel 客戶端安裝與設定
  - 路由規則與安全政策配置
  - 監控與日誌記錄機制

此層級負責處理所有來自網際網路的連線請求，透過 Cloudflare 的全球網路節點提供低延遲、高可用性的存取服務。

HomeLab 與容器編排管理層（中央控制，參照 K8s 運作邏輯）：
- 技術方案：使用 C# 開發自建容器管理與編排系統，參照 Kubernetes（k8s）的運作邏輯與架構設計
- 開發語言：C# / .NET 8.0
- 硬體配置：Raspberry Pi 4（1 台）
- 核心功能模組：
  - 容器管理系統（Container Management）
    - 容器的建立、啟動、停止、刪除等生命週期管理
    - 容器狀態監控與健康檢查
    - 容器資源限制與隔離機制
  - Docker 整合（Docker Integration）
    - 透過 Docker API 與 Docker Engine 進行整合
    - Docker 映像檔管理與版本控制
    - Docker Compose 配置管理
    - 容器網路與儲存卷管理
  - 網路控管系統（Network Control & Security）
    - 防火牆規則設定與管理
    - 網路流量監控與分析
    - 安全政策實作與執行
    - 入侵偵測與防護機制
    - 網路隔離與分段管理
  - 容器編排與調度（參照 K8s 邏輯）
    - 服務發現與負載平衡
    - 資源管理與監控
    - 配置管理與密鑰管理
    - 自動擴展與自我修復機制

此層級建置於 Raspberry Pi 4 上，作為整個系統的核心控制中心，使用 C# 開發自建的容器管理平台，整合 Docker 引擎並提供完整的網路安全控管功能。雖然參照 Kubernetes 的運作邏輯與設計理念，但採用更輕量級且針對樹莓派環境優化的自建實作方案，以適應硬體限制與資源環境。Raspberry Pi 4 的較高效能使其適合擔任此關鍵角色，同時 C# 的跨平台特性與豐富的系統整合能力，使其成為實作此管理層的理想選擇。

應用服務層（AI專案/站臺）：
- 技術方案：容器化應用程式部署
- 硬體配置：Raspberry Pi 3（2 台）
- 功能角色：
  - Horizon Nova 主應用程式
  - AI 模型推理服務（GroundingDINO、CLIP 等）
  - 資料處理與分析
  - 背景任務執行

此層級負責執行實際的業務邏輯與 AI 運算任務。兩台 Raspberry Pi 3 可作為工作節點，透過容器編排系統進行任務分配與負載平衡，提升整體運算能力與容錯性。

架構層級關係（由上而下）：
```
網際網路
    ↓
Cloudflare Tunnel（外網存取層）
    ↓
HomeLab 環境（Pi 4：網路控制與中央管理層）
    ↓
容器編排管理系統（Pi 4：參照 K8s 運作邏輯）
    ↓
工作節點（Pi 3 × 2：應用服務層）
    ↓
容器化應用程式（Horizon Nova + AI 服務）
```

此分層架構設計確保了系統的可擴展性、可維護性與高可用性，每一層級都有明確的職責與介面定義。

三、技術架構分析

3.1 產品技術架構設計

本專案旨在開發一個完整的 HomeLab 容器管理與網路控管系統，採用 C# 開發，部署於 Raspberry Pi 4 上。系統架構設計如下：

**核心架構層級：**

1. **應用程式層（C# 應用程式）**
   - 開發語言：C# / .NET 8.0
   - 架構模式：模組化設計，分離容器管理、Docker 整合、網路控管等核心功能
   - 運行環境：.NET 8.0 Runtime for Linux ARM64

2. **容器管理層（Container Management）**
   - 核心功能：容器生命週期管理、資源監控、狀態追蹤
   - 實作方式：透過 Docker API 與 Docker Engine 進行整合
   - 設計理念：參照 Kubernetes 的運作邏輯，但採用輕量級自建實作

3. **Docker 整合層（Docker Integration）**
   - Docker Engine：作為底層容器運行環境
   - Docker API：透過 RESTful API 與 Docker Engine 通訊
   - 功能範圍：映像檔管理、容器網路、儲存卷管理、Compose 配置

4. **網路控管層（Network Control & Security）**
   - 防火牆管理：規則設定、流量控制、連接追蹤
   - 安全機制：入侵偵測、異常流量分析、安全政策執行
   - 網路隔離：容器網路分段、VLAN 管理、流量路由

3.2 技術選型分析

**開發語言選擇：C# / .NET 8.0**

選擇理由：
- 跨平台支援：.NET 8.0 完整支援 Linux ARM64，可在樹莓派上原生運行
- 豐富的系統整合能力：提供完整的系統 API 存取，便於實作網路控管與容器管理
- 強型別與安全性：降低開發錯誤，提升程式碼品質
- 成熟的生態系統：豐富的 NuGet 套件庫，加速開發進度
- 團隊熟悉度：與現有 Horizon Nova 專案技術棧一致，降低學習成本

**容器技術選擇：Docker**

選擇理由：
- 業界標準：Docker 為最廣泛使用的容器技術，生態系統完整
- ARM64 支援：Docker 完整支援 ARM 架構，可在樹莓派上運行
- API 整合：提供完整的 RESTful API，便於 C# 應用程式整合
- 輕量級：相較於 Kubernetes，Docker 更適合資源受限的樹莓派環境

**網路控管技術選擇：自建系統**

選擇理由：
- 客製化需求：需要針對樹莓派環境與特定使用場景進行優化
- 資源效率：自建系統可針對硬體限制進行最佳化
- 學習價值：透過實作深入理解網路控管與安全機制
- 整合便利：與容器管理系統深度整合，提供統一的控制介面

3.3 技術可行性評估

**硬體資源評估：**

Raspberry Pi 4 的硬體規格足以支援本產品的運行需求：
- CPU：四核心 ARM Cortex-A72，時脈 1.5GHz，足以處理容器管理與網路控管任務
- 記憶體：4GB/8GB LPDDR4，足夠運行 .NET 應用程式、Docker Engine 與系統服務
- 網路：Gigabit 有線網路，提供充足的網路頻寬進行流量控管與分析
- 儲存：128GB SD 卡，足夠儲存應用程式、Docker 映像檔與系統資料

**軟體相容性評估：**

- .NET 8.0：官方完整支援 Linux ARM64，可在樹莓派上原生運行
- Docker：提供 ARM64 版本，完全相容樹莓派硬體
- Linux 系統 API：.NET 提供完整的 Linux 系統呼叫封裝，可實作網路控管功能

**效能考量：**

- 容器管理：輕量級實作設計，避免過度資源消耗
- 網路控管：採用高效能資料結構與演算法，確保低延遲處理
- 資源監控：定期採樣而非持續監控，降低系統負擔

**擴展性評估：**

系統設計採用模組化架構，未來可依需求擴充：
- 支援更多容器管理功能
- 整合更多網路安全機制
- 擴展至多節點管理（管理多台樹莓派）

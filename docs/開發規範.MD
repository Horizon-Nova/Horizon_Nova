========================================
基礎規則（必須嚴格遵守）
========================================

1. 語言要求
   - 禁止使用簡體中文，必須使用繁體中文
   - 所有註解、日誌、文檔、UI 文字都必須使用繁體中文

2. 文檔閱讀（執行前必須完成）
   - 開發前必須閱讀 docs 資料夾中的開發規範和架構文檔
   - 必須理解整體架構設計和職責劃分後才能開始編碼

3. 訊息表示
   - 禁止使用 emoji
   - 必須使用 [成功]、[失敗] 等文字表示

========================================
開發規則（強制執行）
========================================

一、分層架構原則

1. 必須遵守專案的分層架構設計
   - 開發前必須閱讀專案的架構文檔
   - 必須理解各層級的職責劃分
   - 禁止違反分層架構的依賴方向

2. 違反分層的處理方式
   - 發現違反時：立即停止，提出問題並等待確認
   - 修改前：必須先分析所有相關功能，避免產生耦合
   - 修改後：必須驗證所有層級都符合職責

二、程式碼品質要求（核心重點）

1. 避免過度拆分功能

禁止：
- 將一個完整功能拆解成多個小功能後再組合
- 為功能內的操作步驟創建獨立的 function（無需重用的情況下）
- 將流程中的步驟包裝成多個 function 後在主 function 中組合調用

說明：
- 一個完整功能應在同一個 function 內完成所有必要步驟
- 功能內的操作步驟應直接在主 function 內執行，而非為每個步驟創建獨立的 function
- 只有在該步驟需要被多個地方重複使用時，才應將其提取為獨立的 function

原則：
- 功能內的操作步驟應直接在 function 內執行
- 保持功能完整性和連續性
- 只有需要重用或複雜度足夠時才拆分為獨立 function

2. 使用系統邏輯處理（禁止硬編碼判斷疊加）

核心概念：
- 此原則的目的不是單純簡化程式碼，而是正確使用系統提供的機制和邏輯
- 系統已設計好的機制和處理流程，應直接使用而非自行實現
- 讓系統自然處理各種操作和邏輯（如檔案操作、資源管理、驗證、錯誤處理等），而非手動編寫判斷邏輯

禁止：
- 自行實現系統已提供的機制和邏輯（如手動檢查、手動驗證、手動資源管理）
- 使用多層硬編碼的 if-else 判斷疊加，而非使用系統提供的機制
- 為系統已處理的情況添加額外的檢查和判斷（如檢查檔案是否存在、檢查路徑是否有效、檢查資源是否釋放等）
- 手動編寫系統已提供的處理邏輯，忽略系統提供的原生處理方式

必須：
- 使用系統提供的原生機制處理各種操作（如檔案操作、資源管理、驗證、異常處理、空值檢查、數據處理等）
- 檔案操作應使用系統提供的 API，讓系統自動處理路徑檢查、目錄建立、覆蓋保護等邏輯
- 資源管理應使用系統提供的機制，讓系統自動處理資源的建立、使用、釋放等生命週期
- 驗證應使用系統提供的驗證機制，讓系統自然處理驗證錯誤和顯示錯誤訊息
- 錯誤處理應使用系統的異常機制，讓異常自然傳播和處理，而非手動檢查並處理
- 空值檢查應使用系統提供的安全訪問機制，而非手動編寫多層 if 判斷
- 數據處理應使用系統提供的 API 和機制，而非手動實現相同的邏輯

說明：
- 系統處理邏輯不限於錯誤處理，還包括各種操作的處理邏輯（檔案操作、資源管理、驗證等）
- 檔案操作範例：建立檔案功能應直接透過系統 API 和路徑建立，系統會自動處理路徑檢查、目錄建立、覆蓋保護等邏輯，不需要手動檢查檔案是否存在、檢查路徑是否有效、判斷內容是否有資料等
- 應使用系統提供的機制（檔案操作、資源管理、驗證機制、異常處理、空值處理等），讓系統自然處理各種邏輯和邊界情況，而非手動編寫判斷邏輯
- 系統已設計好的機制和處理流程應直接使用，只有在系統不提供相應機制時，才自行實現

原則：
- 優先使用語言/框架/平台提供的原生機制和 API
- 讓系統自然處理各種操作邏輯（檔案操作、資源管理、驗證、錯誤、邊界情況等）
- 使用系統設計的處理流程，而非自己實現相同的邏輯
- 只在系統不提供機制時，才自行實現相應的邏輯

3. 程式碼結構清晰

必須：
- 變數命名語意清晰，禁止使用單字母變數（除了迴圈計數器）
- 註解只解釋「為什麼」，不解釋「做什麼」（程式碼應該自解釋）
- 避免過深的巢狀結構（最多 3 層）
- 相關邏輯放在一起，不相關邏輯分離

4. 命名規則（必須遵守）

禁止：
- 功能名稱使用模糊的動作詞（如 Get、Add、Do、Make 等），無法明確表達功能目的
- 查詢功能命名為 Get...（如 GetUser、GetData 等），應使用 Query 或 Search

必須：
- 功能名稱應明確表達功能目的和行為，而非僅使用動作詞
- 查詢功能應使用 Query 或 Search（如 QueryUser、SearchData）
- 功能名稱應清楚表達功能類型（查詢、建立、刪除、更新等）

說明：
- 功能名稱不是一個動作（Get、Add、Do、Make 等），而是應該明確表達功能目的（Query、Search、Create、Delete、Update 等）
- 查詢功能的正確命名應為 Query 或 Search，而非 Get（如 QueryUser 而非 GetUser）
- 功能名稱應讓其他開發者能夠一眼理解該功能的實際行為和目的

5. 修改前的準備工作（必須遵守）

目的：
- 避免累加功能或重複功能（如 FileServer 內已有建檔流程和功能，FileUIPage 也出現建檔流程和功能）
- 避免錯誤理解需求或偏離功能行為（如需求是搬運 123.txt 內容，撰寫時卻是寫複製 123.txt）
- 確保修正時是修正根本問題，而非添加程式邏輯來達成表面處理

必須：
- 修改前必須查找該功能相關的功能和調用被調用的功能
- 必須先閱讀 UI 和功能描述，理解功能的實際需求和預期行為
- 必須確認需求的具體行為（如搬運、複製、移動等），避免錯誤理解需求
- 必須確認該功能是否已存在於其他模組或位置，避免累加功能或重複功能
- 必須確認修改不會偏離原有邏輯和需求行為
- 必須確認修改不會產生重複邏輯或重複功能
- 修正時不可使用新增方式，必須修正根本問題，而非添加程式邏輯來達成表面處理
- 禁止在未理解功能描述和需求行為的情況下修改 function

6. 自我檢查和修復機制（必須遵守）

1. 完成後的自我檢查
   - 每次完成修改後，必須自動進行自我檢查
   - 檢查是否符合開發規範的所有要求
   - 檢查是否違反基礎規則或開發規則
   - 檢查是否符合需求範圍和預期行為

2. 測試驗證

必須：
   - 每次完成修改後，必須進行測試驗證
   - 驗證功能是否符合需求
   - 驗證是否會產生錯誤或異常
   - 驗證是否符合預期行為

測試記錄（必須遵守）：
   - 每次測試必須記錄測試項目和測試過程到文檔中
   - 必須記錄測試方法（如使用 cmd、shell、Python 等）
   - 必須記錄測試結果（成功或失敗）
   - 必須記錄失敗的原因和使用的測試方式
   - 在下次測試時，必須先查看之前的測試記錄，避免重複使用已經失敗的測試方式
   - 如果某種測試方式失敗，必須標記為失敗，並在下次測試時優先嘗試其他方式
   - 記錄位置：待開發功能.md 或測試項目文檔（根據專案結構決定）

3. 自我修復

必須：
   - 在測試驗證中發現問題時，可以進行自我修復
   - 自我修復必須基於明確的測試結果和清楚的問題原因
   - 自我修復前必須查看之前的測試記錄，避免重複失敗的測試方式
   - 自我修復後必須再次進行測試驗證，確保問題已解決
   - 自我修復後必須更新測試記錄，記錄修復方法和結果

禁止：
   - 如果自我修復後問題仍然存在，必須停止並說明問題，等待開發者確認
   - 如果問題原因不清楚或不確定，禁止自行修復，必須說明問題並等待開發者確認
   - 禁止重複使用已經失敗的測試方式（如第一次使用 cmd 失敗、第二次使用 shell 失敗、第三次又使用 cmd 失敗）

原則：
- 自我修復只能在清楚問題原因和測試驗證的情況下進行
- 自我修復不能違反開發規範的任何原則
- 自我修復後必須驗證修復是否成功
- 如果自我修復不確定或可能產生其他問題，必須停止並等待開發者確認
- 測試記錄必須完整，避免重複失敗的測試方式
- 每次測試前必須查看之前的測試記錄，學習成功的測試方式，避免失敗的方式

========================================
違規處理
========================================

1. 發現違規時
   - 立即停止所有修改
   - 說明違規內容和原因
   - 提出修正方案（僅說明，不執行）
   - 等待開發者明確確認後才能繼續

2. 禁止自行修正
   - 除非開發者明確要求，否則禁止自行修正
   - 禁止在未清楚違規內容和原因的情況下修改
   - 禁止在下次對話中假設開發者已確認而立即修改

========================================
開發規則（強制執行）
========================================

一、思維層規則

1. 底層表徵導向

定義：
- 底層表徵導向是指：開發時先思考系統最底層、最穩定、最可重用的表徵方式與運作規則，再由這些底層結構自然推導出高層功能，而不是直接針對單一功能堆疊流程與判斷
- 此原則的目的不是單純簡化程式碼，而是避免用表面功能思維反覆拼裝邏輯，導致功能只能處理當下情境、無法自然擴展
- 開發者的責任是定義底層表徵、邊界、規則、映射與觸發條件，讓系統依這些基礎結構自行運作並產生結果

思維邏輯：
- 大眾直覺邏輯通常是先想「我要得到什麼結果」，再把流程一步一步寫出來
- 底層表徵導向則是先想「這個系統最底層用什麼形式表達資料、狀態、符號、事件與規則」
- 當底層表徵與規則建立完成後，高層功能應由這套結構推導出來，而不是每新增一種需求就再補一套表面流程
- 這種思維的重點不是刻意使用低階語言，而是優先建立可推導、可組合、可擴展的底層結構

底層：
- 所謂底層，不等於單純追求難懂或貼近硬體，而是指系統中最穩定、最基礎、最能承載多種高層行為的共同結構
- 底層可以是資料結構、狀態模型、符號規則、編碼方式、運算規則、框架原生機制、平台 API 或語言內建能力
- 如果系統、框架、平台已提供成熟機制，應優先直接使用，因為那本身就是現成的底層能力

表徵：
- 表徵是指系統如何一致地表示資料、狀態、符號、事件與規則，讓它們能被統一處理與推導
- 同一套表徵如果足夠穩定，就能讓功能不只解決一個當前問題，而能持續承接後續擴展
- 新需求進來時，應優先判斷是否能透過既有表徵與規則擴展，而不是直接新增一條表面流程

禁止：
- 直接針對表面功能堆疊流程，忽略底層表徵與規則設計
- 為系統已提供的機制額外重寫一套手動處理流程（如手動檢查、手動驗證、手動資源管理）
- 每出現一種新情況就新增一套表面流程，讓系統只能被動追著需求修補

必須：
- 先確認系統最底層的表徵方式、核心規則、狀態流轉與資料映射，再開始設計高層功能
- 使用語言、框架、平台已提供的原生機制作為優先底層能力（如檔案 API、資源管理、驗證、異常機制、空值安全訪問、資料處理 API）
- 設計功能時，優先讓同一套底層結構承接多種高層需求，而非為每個需求各寫一套流程
- 新增功能時，優先擴展既有表徵、規則與映射，只有在現有底層確實不足時，才新增新的底層結構

說明：
- 檔案操作範例：建立檔案功能應直接透過系統 API 和路徑建立，系統會自動處理路徑檢查、目錄建立、覆蓋保護等邏輯，不需要手動檢查檔案是否存在、檢查路徑是否有效、判斷內容是否有資料等
- 計算機範例不應先從「按 1、按 +、按 2、按 =」這種表面流程思考，而應先思考系統如何表徵數字、符號、輸入、狀態與運算規則；當底層表徵一致後，新增符號、文字輸入或其他擴展能力時，只需補充映射與規則，而不是重寫整體流程
- 電梯範例也不應先從每一層移動流程開始堆疊，而應先界定載體、能量、觸發、方向、釋放與回充等底層機制，讓上升、下降與狀態切換成為系統自然運作的結果

原則：
- 優先建立可推導、可組合、可擴展的底層表徵，而不是先寫表面功能流程
- 優先使用現成底層能力，而非自行重寫系統已提供的機制
- 讓高層功能由底層表徵與規則自然產生，而非持續靠表面補丁維持
- 只在現有底層結構無法承接需求時，才新增新的底層結構

二、設計層規則

1. 分層架構原則

必須：
- 開發前必須閱讀專案的架構文檔
- 必須理解各層級的職責劃分
- 禁止違反分層架構的依賴方向

違反分層的處理方式：
- 發現違反時：立即停止，提出問題並等待確認
- 修改前：必須先分析所有相關功能，避免產生耦合
- 修改後：必須驗證所有層級都符合職責

2. 修改前的準備工作

目的：
- 避免累加功能或重複功能（如 FileServer 內已有建檔流程和功能，FileUIPage 也出現建檔流程和功能）
- 避免錯誤理解需求或偏離功能行為（如需求是搬運 123.txt 內容，撰寫時卻是寫複製 123.txt）
- 確保修正時是先定位錯誤、移除錯誤邏輯、回到正確結構，再重新修正，而非直接添加程式邏輯來達成表面處理

必須：
- 修改前必須查找該功能相關的功能和調用被調用的功能
- 必須先閱讀 UI 和功能描述，理解功能的實際需求和預期行為
- 必須確認需求的具體行為（如搬運、複製、移動等），避免錯誤理解需求
- 必須確認該功能是否已存在於其他模組或位置，避免累加功能或重複功能
- 必須確認修改不會偏離原有邏輯和需求行為
- 必須確認修改不會產生重複邏輯或重複功能
- 修正流程必須為：定位錯誤 > 移除錯誤邏輯、錯誤分支、錯誤補丁或錯誤流程 > 回到正確結構 > 重新修正
- 修正時禁止保留舊錯誤邏輯並在其外層疊加新邏輯；不可使用新增方式包覆錯誤，必須直接修正根本問題
- 禁止在未理解功能描述和需求行為的情況下修改 function

三、程式碼品質規則

1. 避免過度拆分功能

禁止：
- 將一個完整功能拆解成多個小功能後再組合
- 為功能內的操作步驟創建獨立的 function（無需重用的情況下）
- 將流程中的步驟包裝成多個 function 後在主 function 中組合調用

說明：
- 一個完整功能應在同一個 function 內完成所有必要步驟
- 功能內的操作步驟應直接在主 function 內執行，而非為每個步驟創建獨立的 function
- 只有在該步驟需要被多個地方重複使用時，才應將其提取為獨立的 function

原則：
- 功能內的操作步驟應直接在 function 內執行
- 保持功能完整性和連續性
- 只有需要重用或複雜度足夠時才拆分為獨立 function

2. 程式碼簡化與分支控制

目的：
- 保持程式碼可讀、可維護、可擴展，避免邏輯只貼合單一情境
- 減少因表面修補造成的複雜度累積與維護成本

禁止：
- 為單一情境持續新增特判與分支，讓邏輯越修越散
- 將本可透過共通規則、資料結構、表徵或平台機制處理的問題，改寫成多段分支流程
- 以語法技巧包裝複雜度，看似簡化但本質仍是重複判斷與重複流程

必須：
- 優先消除重複判斷、重複分支與重複流程
- 優先將共通條件收斂為資料、規則、狀態或既有機制，而非散落在多個位置
- 只保留必要且語意清楚的分支，避免程式碼過度擬合單一案例

說明：
- 少寫分支、避免過度擬合、避免判斷太多，屬於程式碼品質規則，不等於底層表徵導向本身的完整定義
- if-else 不是問題本體；真正的問題是缺乏共通結構時，只能長期靠特判維持功能
- 在明確的狀態分流、錯誤分流或規則分流中使用分支是合理的，但不應讓分支取代結構設計

原則：
- 先判斷能否透過共通結構消除分支，再決定是否保留分支
- 分支用於表達必要差異，不用於長期修補結構缺陷

3. 程式碼結構清晰

必須：
- 變數命名語意清晰，禁止使用單字母變數（除了迴圈計數器）
- 註解只解釋「為什麼」，不解釋「做什麼」（程式碼應該自解釋）
- 避免過深的巢狀結構（最多 3 層）
- 相關邏輯放在一起，不相關邏輯分離

4. 命名規則

禁止：
- 功能名稱使用模糊的動作詞（如 Get、Add、Do、Make 等），無法明確表達功能目的
- 查詢功能命名為 Get...（如 GetUser、GetData 等），應使用 Query 或 Search

必須：
- 功能名稱應明確表達功能目的和行為，而非僅使用動作詞
- 查詢功能應使用 Query 或 Search（如 QueryUser、SearchData）
- 功能名稱應清楚表達功能類型（查詢、建立、刪除、更新等）

說明：
- 功能名稱不是一個動作（Get、Add、Do、Make 等），而是應該明確表達功能目的（Query、Search、Create、Delete、Update 等）
- 查詢功能的正確命名應為 Query 或 Search，而非 Get（如 QueryUser 而非 GetUser）
- 功能名稱應讓其他開發者能夠一眼理解該功能的實際行為和目的

四、驗證與修復規則

1. 自我檢查和修復機制

完成後的自我檢查：
   - 每次完成修改後，必須自動進行自我檢查
   - 檢查是否符合開發規範的所有要求
   - 檢查是否違反基礎規則或開發規則
   - 檢查是否符合需求範圍和預期行為

測試驗證：

必須：
   - 每次完成修改後，必須進行測試驗證
   - 驗證功能是否符合需求
   - 驗證是否會產生錯誤或異常
   - 驗證是否符合預期行為

視覺驗證與截圖：
   - 涉及 UI、樣式、排版、畫面確認時，必須進行視覺驗證
   - 視覺驗證優先使用指定視窗截圖，不可預設使用全螢幕截圖
   - 指定視窗截圖的目的是避免桌面內容、其他視窗、座標偏移或錯誤目標干擾判斷
   - 若指定視窗截圖失敗或邊界不正確，必須先修正截圖方式，再進行畫面確認
   - 只有在明確需要檢查整體桌面互動時，才可使用全螢幕截圖
測試記錄（必須遵守）：
   - 每次測試必須記錄測試項目和測試過程到文檔中
   - 必須記錄測試方法（如使用 cmd、shell、Python 等）
   - 必須記錄測試結果（成功或失敗）
   - 必須記錄失敗的原因和使用的測試方式
   - 在下次測試時，必須先查看之前的測試記錄，避免重複使用已經失敗的測試方式
   - 如果某種測試方式失敗，必須標記為失敗，並在下次測試時優先嘗試其他方式
   - 記錄位置：待開發功能.md 或測試項目文檔（根據專案結構決定）

自我修復：

必須：
   - 在測試驗證中發現問題時，可以進行自我修復
   - 自我修復必須基於明確的測試結果和清楚的問題原因
   - 自我修復前必須查看之前的測試記錄，避免重複失敗的測試方式
   - 自我修復後必須再次進行測試驗證，確保問題已解決
   - 自我修復後必須更新測試記錄，記錄修復方法和結果

禁止：
   - 如果自我修復後問題仍然存在，必須停止並說明問題，等待開發者確認
   - 如果問題原因不清楚或不確定，禁止自行修復，必須說明問題並等待開發者確認
   - 禁止重複使用已經失敗的測試方式（如第一次使用 cmd 失敗、第二次使用 shell 失敗、第三次又使用 cmd 失敗）

原則：
- 自我修復只能在清楚問題原因和測試驗證的情況下進行
- 自我修復不能違反開發規範的任何原則
- 自我修復後必須驗證修復是否成功
- 如果自我修復不確定或可能產生其他問題，必須停止並等待開發者確認
- 測試記錄必須完整，避免重複失敗的測試方式
- 每次測試前必須查看之前的測試記錄，學習成功的測試方式，避免失敗的方式

========================================
違規處理
========================================

1. 發現違規時
   - 立即停止所有修改
   - 說明違規內容和原因
   - 提出修正方案（僅說明，不執行）
   - 等待開發者明確確認後才能繼續

2. 禁止自行修正
   - 除非開發者明確要求，否則禁止自行修正
   - 禁止在未清楚違規內容和原因的情況下修改
   - 禁止在下次對話中假設開發者已確認而立即修改

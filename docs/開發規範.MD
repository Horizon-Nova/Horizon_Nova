開發規範

========================================
強制性規則（必須嚴格遵守）
========================================

1. 語言要求
   - 禁止使用簡體中文，必須使用繁體中文
   - 所有註解、日誌、文檔、UI 文字都必須使用繁體中文

2. 文檔閱讀（執行前必須完成）
   - 開發前必須閱讀 docs 資料夾中的開發規範和架構文檔
   - 必須理解整體架構設計和職責劃分後才能開始編碼

3. 訊息表示
   - 禁止使用 emoji
   - 必須使用 [成功]、[失敗] 等文字表示

========================================
程式碼結構要求（強制執行）
========================================

分層架構原則（必須嚴格遵守）

1. 必須遵守專案的分層架構設計
   - 開發前必須閱讀專案的架構文檔
   - 必須理解各層級的職責劃分
   - 禁止違反分層架構的依賴方向

2. 違反分層的處理方式
   - 發現違反時：立即停止，提出問題並等待確認
   - 修改前：必須先分析所有相關功能，避免產生耦合
   - 修改後：必須驗證所有層級都符合職責

========================================
程式碼品質要求（強制執行，核心重點）
========================================

1. 避免死邏輯（1+1 邏輯）- 絕對禁止

禁止寫法（錯誤範例）：
if (field.equals("Part Number")) {
    value = data.getPartNumber();
} else if (field.equals("Lot")) {
    value = data.getLot();
} else if (field.equals("Press")) {
    value = data.getPress();
}

必須寫法（正確範例）：
value = data.getField(field); // 統一處理所有欄位

原則：
- 禁止硬編碼特定情況的判斷
- 必須使用抽象、通用的邏輯
- 如果有多個相似的 if-else，必須重構為抽象邏輯
- 使用 Map、策略模式、或統一介面來處理多種情況

2. 函式單一責任 - 絕對要求

禁止：
- 一個函式做多件事
- 無意義的代理函式（僅包裝同一呼叫，沒有額外邏輯）
- 函式名稱與實際行為不符
- 函式過長（超過 50 行必須拆分）

必須：
- 每個函式只做一件事
- 函式名稱清楚表達行為（動詞開頭，語意明確）
- 複雜邏輯拆分成多個小函式
- 函式長度控制在合理範圍內

3. 表達式優先 - 必須遵守

禁止（多餘變數和多層判斷）：
String result = null;
if (data != null) {
    if (data.getField() != null) {
        result = data.getField().trim();
    }
}
return result;

必須（使用原生安全訪問和就地運算）：
return data != null ? data.getField() : null;
或使用 Optional：
return Optional.ofNullable(data).map(d -> d.getField()).orElse(null);

原則：
- 使用語言/框架原生的空值預設/安全訪問
- 就地運算，避免多餘的中間變數
- 避免多層巢狀判斷
- 優先使用三元運算符、Optional、Stream API 等現代語法

4. 程式碼結構清晰

必須：
- 變數命名語意清晰，禁止使用單字母變數（除了迴圈計數器）
- 註解只解釋「為什麼」，不解釋「做什麼」（程式碼應該自解釋）
- 避免過深的巢狀結構（最多 3 層）
- 相關邏輯放在一起，不相關邏輯分離

========================================
需求遵守規則（絕對命令，最高優先級）
========================================

1. 嚴格遵守需求（絕對不可違反）
   - 禁止自行擴展功能
   - 禁止修改需求範圍外的內容
   - 禁止「優化」未要求的部分
   - 禁止添加「可能有用」的功能
   - 禁止修改未明確要求修改的程式碼
   - 只能實現開發者明確要求的功能

2. 遇到問題時的處理流程（必須嚴格執行）
   1. 立即停止所有修改
   2. 明確說明問題點和原因
   3. 提出可能的解決方案（僅說明，不執行）
   4. 等待開發者明確確認後才能繼續
   5. 禁止自行判斷或猜測開發者意圖

3. 錯誤處理原則（絕對遵守）
   - 一旦發現錯誤邏輯，立即停止
   - 必須恢復到原本程式狀態
   - 禁止在錯誤邏輯上延伸修改
   - 禁止「修復」未要求修復的問題

4. 需求理解確認
   - 如果需求不明確，必須先詢問開發者
   - 禁止自行解釋或推測需求
   - 禁止「我認為應該這樣做」的判斷

========================================
架構設計原則（強制執行）
========================================

避免過耦合

1. 修改前必須做的事
   - 反推所有相關功能
   - 分析功能之間的依賴關係
   - 確認修改不會產生耦合

2. 功能獨立性
   - 必須保持功能之間的獨立性
   - 禁止讓不同功能過度耦合
   - 修改一個功能時，不應影響其他功能

3. 依賴方向
   - 必須遵守專案定義的依賴方向
   - 禁止反向依賴或跨層依賴
   - 必須在架構文檔中確認依賴關係


========================================
檢查清單（每次修改前必須確認）
========================================

- 是否閱讀了相關文檔和架構設計？
- 是否符合專案的分層架構要求？
- 是否避免了死邏輯（1+1 邏輯）？
- 函式是否單一責任？
- 是否使用了表達式優先（避免多餘變數和多層判斷）？
- 是否嚴格遵守了需求範圍（未擴展功能）？
- 是否會產生過耦合？
- 是否使用了繁體中文？

========================================
違規處理
========================================

- 發現違規：立即停止，說明違規內容
- 等待確認：提出修正方案，等待開發者同意
- 禁止自行修正：除非開發者明確要求

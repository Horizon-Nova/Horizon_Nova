架構規則

========================================
分層架構
========================================

UI 層
  ↓ 調用
服務層
  ↓ 調用
工具層
  ↓ 調用（如需要）
Core 層

========================================
UI 層
========================================

職責：
- 只負責調用服務層，不包含任何業務邏輯
- 負責 UI 元件的建立、佈局、事件綁定
- ViewModel 的職責和 JS 一樣屬於做動畫（防抖、動畫過渡、狀態管理）

禁止：
- 禁止包含業務邏輯
- 禁止包含判斷邏輯
- 禁止直接調用工具層
- 禁止直接調用 Core 層（除了 ViewModel 中的動畫相關邏輯）

允許：
- UI 元件的事件處理（僅調用服務層）
- ViewModel 中的動畫相關邏輯（防抖、動畫過渡）
- ViewModel 中的狀態管理（Property、ObservableList）

========================================
服務層
========================================

職責：
- 屬於橋梁，不做任何邏輯或判斷
- 純串接服務和 return
- 作為 UI 層與工具層之間的橋樑

規則：
- 函式不超過 3 行
- 禁止包含業務邏輯
- 禁止包含判斷邏輯
- 禁止包含迴圈邏輯
- 只能進行簡單的參數傳遞和結果返回

========================================
工具層
========================================

職責：
- 做全部邏輯和判斷
- 包含完整的業務邏輯實現
- 實現具體的技術功能
- 處理複雜的判斷、迴圈、異常處理等邏輯

規則：
- 可以包含完整的業務邏輯
- 可以包含判斷邏輯
- 可以包含迴圈邏輯
- 可以包含異常處理
- 函式長度根據實際需求決定（但需遵守函式單一責任原則）

========================================
Core 層
========================================

職責：
- 屬於核心技術
- 當一個功能被多個調用就屬於核心

規則：
- 提供被多個地方共用的核心功能
- 例如：PropertiesCore（被多個服務和工具層調用）、ServiceContainer（統一管理服務）

========================================
依賴方向規則
========================================

- UI 層只能調用服務層
- 服務層只能調用工具層
- 工具層可以調用 Core 層
- 禁止反向依賴
- 禁止跨層依賴（例如：UI 層直接調用工具層）

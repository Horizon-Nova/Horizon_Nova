# 服裝 AI 系統設計說明

## 系統目的
為使用者提供智能服裝管理系統，透過 AI 技術自動識別、分類和生成服裝商品圖，協助使用者管理和組織個人服裝收藏。

---

## 使用者角色

### 主要使用者
- **個人使用者（衣櫃管理）**
  - 上傳個人服裝照片
  - 查看和管理已識別的服裝
  - 查看 AI 生成的商品圖（用於展示或參考）

### 系統功能使用者
- **後台管理員**
  - 管理系統設定
  - 監控 AI 處理狀態
  - 查看處理記錄

## 操作流程

### 1. 上傳與識別流程（Wardrobe 頁面）

#### 步驟說明
1. **使用者上傳圖片**
   - 拖曳圖片或選擇檔案
   - 支援多檔案上傳
   - 支援格式：jpg, jpeg, png, gif, bmp, webp

2. **系統自動處理**
   ```
   上傳圖片 
   → 保存原圖（原圖/{日期}/{guid}_{檔名}）
   → 物件偵測（使用 GroundingDINO 模型）
   → 裁剪識別結果（辨識/{日期}/{類別}/{guid}_{類別}_{序號}.png）
   → 自動生成商品圖（商品圖/{日期}/product_{辨識檔名}.png）
   ```

3. **結果展示**
   - 顯示識別出的服裝項目
   - 每個項目包含：類別、信心度、裁剪後圖片路徑

#### 檔案組織結構
```
storage/ObjectDetection/
├── 原圖/
│   └── 2025_10_31/
│       └── {guid}_{原始檔名}.png
├── 辨識/
│   └── 2025_10_31/
│       ├── clothes/
│       │   └── {guid}_clothes_{序號}.png
│       ├── jacket/
│       │   └── {guid}_jacket_{序號}.png
│       └── pants/
│           └── {guid}_pants_{序號}.png
└── 商品圖/
    └── 2025_10_31/
        └── product_{辨識檔名}.png  (由 DALL-E 生成)
```

### 2. 商品圖生成流程

#### 自動生成（上傳時自動執行）
- 上傳圖片並完成辨識後，**自動為每個辨識結果生成商品圖**
- 流程：辨識完成 → 立即呼叫 DALL-E → 保存商品圖

#### 手動重新生成（透過 GenerateProductImages API）
1. **傳入辨識檔名**
   - 格式：`{日期}/{類別}/{檔名}`
   - 例如：`2025_10_31/clothes/abc123_clothes_0.png`
   - **重要**：只傳入檔名，不掃描整個資料夾，避免費用浪費

2. **系統根據檔名查找**
   ```
   傳入辨識檔名列表
   → 直接根據檔名路徑查找檔案（不掃描資料夾）
   → 根據類別生成 Prompt
   → 呼叫 DALL-E 3 API
   → 下載生成的圖片
   → 保存到商品圖/{日期}/product_{檔名}.png
   ```

3. **生成結果**
   - 成功：返回商品圖路徑
   - 失敗：返回錯誤訊息

#### DALL-E Prompt 模板
```
"A clean, professional product photo of a {類別} on a transparent background, 
studio lighting, minimalist composition, high quality commercial photography"
```

### 3. 衣櫃管理流程

#### 查看服裝列表
- 從「商品圖」資料夾載入所有已生成的商品圖
- 顯示統計資訊：
  - 總服裝數
  - 本月上傳數量
  - 總大小
  - 最近更新數量

#### 刪除服裝
- 使用者可以刪除不需要的商品圖
- 刪除操作只針對「商品圖」資料夾

## 需要記錄的資料

### 1. 檔案記錄（檔案系統）

#### 原圖記錄
- **路徑**：`原圖/{日期}/{guid}_{原始檔名}`
- **命名規則**：`{guid}_{原始上傳檔名}`
- **用途**：保留原始上傳圖片，供後續處理參考

#### 辨識結果記錄
- **路徑**：`辨識/{日期}/{類別}/{guid}_{類別}_{序號}.{副檔名}`
- **命名規則**：
  - `{guid}`：該批次上傳的唯一識別碼
  - `{類別}`：AI 識別出的服裝類別（clothes, jacket, pants 等）
  - `{序號}`：同一圖片中多個相同類別的編號
  - **不包含座標**：座標資訊不在檔名中（設計中無需從檔名讀取座標）
- **用途**：保存裁剪後的單一服裝物件圖片

#### 商品圖記錄
- **路徑**：`商品圖/{日期}/product_{辨識檔名}`
- **命名規則**：`product_{辨識檔名}`（保留原始辨識檔名，加上 product_ 前綴）
- **用途**：DALL-E 生成的商業級商品展示圖

#### 建議記錄的資料

## 資料記錄方式

### 目前實作方式（檔案系統）

#### 檔案命名規範
1. **原圖**：`{guid}_{原始檔名}`
   - 確保同一批次上傳的檔案可以追溯
   - guid 用於關聯同一批次的處理結果

2. **辨識結果**：`{guid}_{類別}_{序號}.{副檔名}`
   - 簡化檔名，不包含座標（設計中無需從檔名讀取座標）
   - 可以從檔名還原基本識別資訊

3. **商品圖**：`product_{辨識檔名}`
   - 可以追溯原始辨識結果
   - product_ 前綴用於區分是否為商品圖

#### 費用控制設計
1. **避免掃描資料夾**
   - 生成商品圖時，只傳入檔名列表，系統直接根據路徑查找
   - 不掃描整個「辨識」或「商品圖」資料夾，避免大量檔案讀取操作

2. **自動生成時機**
   - 僅在上傳識別完成後立即生成
   - 不重複生成已存在的商品圖

3. **手動重新生成**
   - 使用者明確指定要重新生成的檔名
   - 系統只處理指定的檔案，不掃描全部

#### 資料夾組織
- **按日期分類**：所有資料夾以 `yyyy_MM_dd` 格式命名
- **按類別分類**：辨識結果按類別（clothes, jacket, pants 等）分資料夾
- **結構化查詢**：`LoadWardrobeList()` 方法掃描「商品圖」資料夾，建立列表

### 建議改進（資料庫記錄）

#### 為什麼需要資料庫記錄
1. **追蹤處理狀態**：知道哪些圖片已處理、哪些失敗
2. **查詢效能**：避免每次都要掃描檔案系統
3. **統計分析**：可以統計處理量、成功率等
4. **錯誤追蹤**：記錄失敗原因，便於除錯
5. **使用者關聯**：知道哪些服裝屬於哪個使用者

#### 建議的資料庫結構
```sql
-- 上傳記錄
CREATE TABLE clothing_uploads (
    id UUID PRIMARY KEY,
    user_id INT,
    original_filename VARCHAR(255),
    file_path VARCHAR(500),
    file_size BIGINT,
    upload_time TIMESTAMP,
    status VARCHAR(20)  -- pending, processing, completed, failed
);

-- 識別結果
CREATE TABLE clothing_detections (
    id UUID PRIMARY KEY,
    upload_id UUID REFERENCES clothing_uploads(id),
    category VARCHAR(50),
    confidence DECIMAL(5,2),
    x INT,
    y INT,
    width INT,
    height INT,
    cropped_path VARCHAR(500),
    detected_time TIMESTAMP
);

-- 商品圖生成記錄
CREATE TABLE product_image_generations (
    id UUID PRIMARY KEY,
    detection_id UUID REFERENCES clothing_detections(id),
    prompt TEXT,
    api_response_time_ms INT,
    status VARCHAR(20),  -- success, failed
    error_message TEXT,
    product_image_path VARCHAR(500),
    generated_time TIMESTAMP
);
```

## 系統限制與注意事項

### 目前限制
1. **無資料庫記錄**：所有資料僅存在檔案系統中
2. **無使用者隔離**：所有上傳的檔案都在同一資料夾中
3. **無處理佇列**：同步處理，大量上傳可能導致延遲
4. **無錯誤重試機制**：DALL-E API 失敗後不會自動重試

### 重要設計決策
1. **自動生成商品圖**：
   - 上傳時自動完成：識別 → 裁剪 → 生成商品圖
   - 確保每次上傳都產生完整的處理結果
   - 手動重新生成僅用於失敗重試或重新生成需求

2. **費用控制策略**：
   - **LoadWardrobeList**：這是唯一需要掃描資料夾的場景（用戶查看衣櫃）
   - **GenerateProductImagesByFileNames**：根據明確的檔名列表直接查找，不掃描
   - **ProcessSingleImage**：自動生成時使用已建立的檔案路徑，無需查找

3. **檔案命名規範**：
   - 使用 guid 關聯同一批次的處理結果
   - 檔名簡化：不包含座標（設計中無需從檔名讀取座標）
   - 可以從檔名還原基本識別資訊（guid、類別、序號）

4. **資料夾結構**：
   - 按日期分類，便於管理和清理
   - 按類別分類，便於查找同類服裝
   - 結構化路徑便於直接存取，避免大量掃描
